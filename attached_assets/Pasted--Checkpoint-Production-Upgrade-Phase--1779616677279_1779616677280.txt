1. الهدف التنفيذي
==================================================

الهدف هو تجهيز وربط Supabase/Auth/RLS بشكل آمن للوصول لاحقاً إلى Production Ready.

المطلوب:

1. فحص ملفات Supabase الحالية:
   - SUPABASE_SCHEMA.sql
   - RLS_POLICIES.sql
   - ENV_EXAMPLE.md
   - .env.example

2. التأكد من متغيرات البيئة:
   - VITE_SUPABASE_URL
   - VITE_SUPABASE_ANON_KEY

3. ممنوع استخدام:
   - service_role key داخل الواجهة.
   - أي مفتاح إداري client-side.

4. تفعيل Supabase client بشكل آمن.

5. إنشاء أو تفعيل Auth Service:
   - تسجيل دخول.
   - تسجيل خروج.
   - قراءة session.
   - حماية routes الإدارية.
   - demo mode fallback فقط عند غياب المفاتيح.

6. ربط profiles بـ auth.users.

7. تفعيل أو تجهيز:
   - roles.
   - permissions.
   - user_roles.
   - admin_users.

8. تفعيل RLS فعلياً في Supabase إذا كانت البيئة والمفاتيح متاحة.

9. اختبار أن:
   - المستخدم العادي لا يدخل /admin.
   - admin يستطيع الدخول للأقسام الإدارية.
   - super_admin لديه صلاحيات أوسع.
   - المستخدم يرى بياناته فقط.
   - البيانات العامة تظهر حسب status/is_active.

10. الحفاظ على كل وظائف التطبيق السابقة.

11. تحديث التوثيق والحكم النهائي بصدق.

12. ما استراتيجية النقل الآمن إلى Supabase؟

بعد التشخيص:
- إذا مفاتيح Supabase غير موجودة:
  لا تنفذ ربطاً وهمياً.
  حدّث التوثيق فقط، واترك التطبيق Publishable Preview.
- إذا المفاتيح موجودة:
  نفّذ الربط تدريجياً وبأقل تغيير آمن.
- لا تكسر fallback الحالي حتى يعمل Supabase بثبات.

==================================================
5. قاعدة التنفيذ الآمن
==================================================

نفذ بأقل تغيير آمن.

القواعد:
- اجعل Supabase layer خلف service interface.
- لا تجعل المكونات تعرف هل المصدر fallback أو Supabase.
- حافظ على fallback عند غياب المفاتيح.
- استخدم anon key فقط في الواجهة.
- لا تستخدم service_role.
- RLS هو خط الدفاع الحقيقي.
- route guard في الواجهة ليس بديلاً عن RLS.
- أي عملية admin حساسة يجب أن تمر عبر سياسات RLS أو توثق أنها تحتاج backend/edge function لاحقاً.
- لا تكسر:
  - الرئيسية.
  - التقويم.
  - المال.
  - ستوري اليوم.
  - الإشعارات.
  - المراكز.
  - حسابي.
  - /admin.
  - Theme Engine.

7. المتطلبات التفصيلية
==================================================

أ. Environment:
- استخدم VITE_SUPABASE_URL.
- استخدم VITE_SUPABASE_ANON_KEY.
- لا تستخدم service_role.
- حدّث ENV_EXAMPLE.md عند الحاجة.
- حدّث .env.example بدون قيم حقيقية.

ب. Supabase Client:
- إنشاء/مراجعة supabase client.
- قراءة env بأمان.
- fallback واضح عند غياب env.
- رسائل خطأ تطويرية لا تظهر للمستخدم العادي.

ج. Auth:
- login.
- logout.
- session.
- auth state listener إن مناسب.
- demo mode fallback عند غياب Supabase.
- لا تدّع production auth إذا demo.

د. Route Guard:
- /admin محمي.
- admin routes تتحقق من role.
- user العادي لا يدخل admin.
- unauthenticated user يوجّه إلى login أو demo notice حسب الوضع.

هـ. Roles/Permissions:
- user.
- admin.
- super_admin.
- content_manager.
- finance_manager.
- permissions mapping.

و. RLS:
- المستخدم يرى بياناته فقط.
- admin/super_admin يدير البيانات العامة والإدارية.
- content_manager يدير الرسائل والأخبار والوظائف وقوالب الستوري.
- finance_manager يدير الأحداث المالية.
- audit_logs قراءة إدارية فقط.
- public active data readable حسب الحاجة.

ز. Data Migration Readiness:
- لا تمسح بيانات حالية.
- إن احتجت seed، استخدم upsert آمن.
- وثّق ما يحتاج تنفيذ يدوي في Supabase SQL Editor.
- لا تنقل بشكل destructive.

ح. Security:
- فحص عدم وجود secrets.
- فحص عدم وجود service_role.
- فحص عدم وجود مفاتيح داخل الملفات العامة.
- تحديث QA_REPORT.
