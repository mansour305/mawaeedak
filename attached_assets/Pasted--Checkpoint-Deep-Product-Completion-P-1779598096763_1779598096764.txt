==================================================
1. الهدف التنفيذي
==================================================

حوّل التقويم من صفحة عرض فقط إلى وحدة مواعيد شخصية تعمل فعلياً.

المطلوب تنفيذ أو التحقق من التالي:

1. عرض المواعيد الشخصية في صفحة التقويم.
2. إضافة موعد جديد.
3. تعديل موعد موجود.
4. حذف موعد موجود مع Confirm Dialog.
5. تصنيف الموعد.
6. تحديد لون أو نوع الموعد إن كان موجوداً في البنية.
7. تحديد تاريخ ووقت الموعد.
8. إضافة ملاحظة.
9. البحث في المواعيد إن كانت البنية الحالية تدعم ذلك.
10. فلترة حسب التصنيف إن كانت البنية الحالية تدعم ذلك.
11. ظهور المواعيد القادمة في الصفحة الرئيسية.
12. بقاء المواعيد بعد reload.
13. مرور كل العمليات عبر Data Layer / service layer، وليس state محلي داخل component فقط.
14. عدم كسر أي بيانات حالية.
15. تحديث QA_REPORT بنتيجة المرحلة.

لا تنفذ أي ميزة خارج التقويم والمواعيد الشخصية إلا إذا كانت ضرورية مباشرة لإكمال سلسلة أثر الموعد.

==================================================
3. ممنوعات صارمة
==================================================

ممنوع:
- لا تعيد بناء المشروع من البداية.
- لا تغيّر Routing.
- لا تغيّر API إلا إذا كان هناك endpoint خاص بالمواعيد مكسور ويحتاج إصلاحاً محدوداً.
- لا تغيّر Data Layer جذرياً.
- لا تغيّر الثيم أو Visual Design إلا لإصلاح كسر ناتج عن نموذج أو نافذة.
- لا تضف Supabase الآن.
- لا تضف Firebase الآن.
- لا تضغط Publish.
- لا تجعل الموعد محفوظاً داخل component state فقط.
- لا تستخدم hardcoded appointments داخل الصفحة.
- لا تترك زر إضافة أو تعديل أو حذف بلا وظيفة.
- لا تدّع نجاح اختبار لم تشغّله.
- لا تغيّر الحكم إلى Production Ready.

==================================================
4. بروتوكول التشخيص قبل التنفيذ
==================================================

قبل التعديل، افحص:

1. أين صفحة التقويم؟
2. أين يتم تعريف appointment types؟
3. أين service layer الخاص بالمواعيد؟
4. هل توجد fallback service للمواعيد؟
5. هل يوجد API endpoint للمواعيد؟
6. هل الصفحة الرئيسية تقرأ المواعيد القادمة من نفس المصدر؟
7. هل بيانات المواعيد تبقى بعد reload؟
8. هل يوجد نموذج إضافة موعد؟
9. هل يوجد تعديل وحذف؟
10. هل توجد أخطاء حالية في console أو API؟

بعد التشخيص:
- إذا كانت الوظيفة موجودة جزئياً، أكمل الناقص فقط.
- إذا كانت موجودة لكنها لا تمر عبر Data Layer، أصلح الربط.
- إذا كانت غير موجودة، نفذ الحد الأدنى الكامل لهذه المرحلة.
- لا تلمس أجزاء خارج النطاق.

==================================================
5. قاعدة الإصلاح الآمن
==================================================

نفذ بأقل تغيير آمن.

القواعد:
- واجهة التقويم تقرأ من appointment service.
- إضافة الموعد تمر عبر service.
- تعديل الموعد يمر عبر service.
- حذف الموعد يمر عبر service.
- الصفحة الرئيسية تقرأ المواعيد القادمة من نفس مصدر البيانات.
- البيانات تبقى بعد reload عبر المصدر الحالي المستخدم في المشروع.
- أي حذف يحتاج Confirm Dialog.
- أي عملية نجاح أو فشل تعرض feedback واضحاً.
- لا تكسر RTL.
- لا تكسر Mobile layout.
- لا تكسر Visual Polish المعتمد.

6. مصدر الحقيقة
==================================================

مصدر الحقيقة لهذه المرحلة:

appointments عبر Data Layer / service layer الحالي.

ممنوع:
- تكرار المواعيد داخل components.
- وجود نسخة في التقويم ونسخة مختلفة في الرئيسية.

- فقدان البيانات بعد reload إذا كان fallback الحالي يدعم الحفظ.

المطلوب:
- التقويم والرئيسية يعتمدان على نفس appointment source.
- أي موعد جديد يظهر في التقويم والرئيسية.
- أي تعديل يظهر في التقويم والرئيسية.
- أي حذف يختفي من التقويم والرئيسية.

==================================================

نفذ أو تحقق من:

- عرض قائمة أو شبكة مواعيد واضحة.

- الموعد المحذوف يختفي من الرئيسية.
- الموعد المعدل يظهر بتعديله.

ج. Data Layer:
- createAppointment
- updateAppointment
- deleteAppointment
- listAppointments
أو ما يعادلها حسب بنية المشروع.

د. UX:
- Toast أو رسالة نجاح عند الإضافة.
- Toast أو رسالة نجاح عند التعديل.
- Toast أو رسالة نجاح عند الحذف.
- رسالة خطأ عند فشل الحفظ.
- RTL سليم.
- Mobile-first سليم.

8. الاختبارات الإلزامية
==================================================

11. تحقق أن التعديل ظهر في الرئيسية.
12. أعد تحميل الصفحة reload.
13. تحقق أن الموعد بقي موجوداً.
14. احذف الموعد مع Confirm Dialog.
15. تحقق أنه اختفى من التقويم.
16. تحقق أنه اختفى من الرئيسية.
17. تحقق أنه لا توجد أخطاء API 500.
18. تحقق أنه لا توجد شاشة بيضاء.
