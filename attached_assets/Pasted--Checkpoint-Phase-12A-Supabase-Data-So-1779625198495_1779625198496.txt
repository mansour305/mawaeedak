استمر من آخر Checkpoint. لا تعيد البناء من البداية.

المرحلة الحالية:
Phase 12A
Supabase Data Source Mapping + Migration Readiness Gate

الدور:
تصرف كفريق Database Migration Architect + Backend Engineer + Data Integrity Auditor + QA Lead.
مهمتك الآن ليست نقل البيانات فوراً، وليست تعديل الواجهة، وليست حذف أو استبدال المصدر الحالي.
مهمتك فقط فحص مصادر البيانات الحالية، ومطابقتها مع جداول Supabase، وتجهيز خطة Migration آمنة قابلة للتنفيذ لاحقاً.

==================================================
0. السياق الحالي
==================================================

المشروع وصل إلى:
Full Supabase Connected Preview

المعتمد حالياً:
- Supabase env موجود.
- Supabase client يعمل.
- 19/19 جداول موجودة.
- RLS مفعّل.
- Auth يعمل.
- /admin محمي عبر Supabase Auth.
- hrq@hotmail.com دخل كـ super_admin.
- build ناجح.
- typecheck ناجح.
- QA_REPORT محدث.
- لا service_role في الواجهة.
- لا أسرار hardcoded.

القيد الحالي:
- بيانات التطبيق التشغيلية ما زالت في PostgreSQL/API الحالي.
- جداول Supabase التشغيلية موجودة لكنها ليست مصدر الحقيقة بعد.
- لا نريد نقل بيانات عشوائي أو destructive migration.

==================================================
1. الهدف التنفيذي
==================================================

جهّز المشروع لمرحلة نقل البيانات إلى Supabase بأمان.

المطلوب:
1. افحص جميع مصادر البيانات الحالية في التطبيق.
2. حدّد ما يأتي من PostgreSQL/API الحالي.
3. حدّد ما يأتي من localStorage.
4. حدّد ما يأتي من Supabase حالياً.
5. طابق كل مصدر بيانات مع جدول Supabase المقابل.
6. حدّد الجداول التي تحتاج seed أو migration.
7. حدّد الجداول التي تحتاج user_id أو ownership.
8. حدّد الجداول العامة admin_managed.
9. حدّد أي اختلاف بين schema الحالي وSupabase schema.
10. لا تنقل أي بيانات الآن.
11. لا تغيّر مصدر الحقيقة الآن.
12. لا تكسر API الحالي.
13. أنشئ خطة Migration تفصيلية.
14. أنشئ ملف SQL/Markdown تحضيري فقط إن احتجت.
15. حدّث QA_REPORT بالحالة الحقيقية.
