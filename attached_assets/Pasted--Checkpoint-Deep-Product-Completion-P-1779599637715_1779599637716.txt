==================================================
1. الهدف التنفيذي
==================================================

حوّل مركز الإشعارات من عرض بسيط إلى وحدة إشعارات داخلية عملية تعمل فعلياً.

المطلوب تنفيذ أو التحقق من التالي:

1. مركز الإشعارات يعرض سجل إشعارات داخلي.
2. الإشعارات تأتي من Data Layer / service layer، وليس من نصوص ثابتة داخل component.
3. إرسال إشعار داخلي من لوحة المالك /admin.
4. ظهور الإشعار المرسل فوراً في مركز إشعارات المستخدم.
5. حالة مقروء / غير مقروء.
6. عداد إشعارات غير مقروءة.
7. تغيير حالة الإشعار إلى مقروء.
8. انعكاس تغيير الحالة على العداد.
9. حذف إشعار أو أرشفته إن كانت البنية الحالية تدعم ذلك.
10. بقاء الإشعارات وحالاتها بعد reload.
11. إعدادات تفعيل/إيقاف أنواع الإشعارات من حسابي إن كانت البنية الحالية تدعم ذلك.
12. توثيق أن Push/Firebase غير مفعل الآن، وأن المنفذ هو إشعار داخلي فقط.
13. تحديث QA_REPORT بنتيجة المرحلة.

- وثّق أن Push مؤجل لإصدار لاحق.

==================================================
3. ممنوعات صارمة
==================================================

ممنوع:
- لا تعيد بناء المشروع من البداية.
- لا تغيّر Routing.
- لا تغيّر API إلا إذا كان endpoint خاص بالإشعارات مكسوراً ويحتاج إصلاحاً محدوداً.
- لا تغيّر Data Layer جذرياً.
- لا تغيّر Theme العام إلا لإصلاح كسر بسيط داخل صفحة الإشعارات.
- لا تضف Supabase الآن.
- لا تضف Firebase الآن.
- لا تضغط Publish.
- لا تدّع وجود Push Notification حقيقي إذا لم يُنفذ.
- لا تجعل الإشعارات hardcoded داخل component.
- لا تجعل زر إرسال الإشعار في /admin بلا أثر.
- لا تجعل زر مقروء/غير مقروء بلا أثر.
- لا تدّع نجاح build أو typecheck إن لم يتم تشغيلهما.
- لا تغيّر الحكم إلى Production Ready.

==================================================
4. بروتوكول التشخيص قبل التنفيذ
==================================================

قبل التعديل، افحص:

1. أين صفحة مركز الإشعارات؟
2. هل توجد notification service أو Data Layer خاص بالإشعارات؟
3. هل توجد API endpoints للإشعارات؟
4. هل توجد جداول أو models للإشعارات؟
5. هل /admin يحتوي قسم إرسال إشعار؟
6. هل زر الإرسال يعمل فعلياً؟
7. هل الإشعار يظهر في واجهة المستخدم؟
8. هل توجد حالة read/unread؟
9. هل يوجد unread count؟
10. هل العدّاد يظهر في الهيدر أو الرئيسية أو تبويب الإشعارات؟
11. هل الحالة تبقى بعد reload؟
12. هل حسابي يحتوي إعدادات إشعارات؟
13. هل توجد أخطاء API أو console حالية؟

بعد التشخيص:
- إذا كانت الوظيفة موجودة جزئياً، أكمل الناقص فقط.
- إذا كانت موجودة لكنها تعتمد على نص ثابت، اربطها بالـ Data Layer.
- إذا كان إرسال /admin شكلياً، اجعله ينتج إشعاراً يظهر للمستخدم.
- لا تلمس أجزاء خارج النطاق.

==================================================
5. قاعدة الإصلاح الآمن
==================================================

نفذ بأقل تغيير آمن.

القواعد:
- مركز الإشعارات يقرأ من notification service.
- /admin يرسل الإشعار عبر نفس service.
- حالة read/unread تمر عبر service.
- العداد يحسب من نفس مصدر البيانات.
- لا تكرر الإشعارات داخل components.
- لا تستخدم source منفصل للـ admin وsource آخر للمستخدم.
- أي عملية نجاح أو فشل تعرض feedback واضحاً.
- لا تكسر RTL.
- لا تكسر Mobile layout.
- لا تكسر Visual Polish المعتمد.
- لا تكسر Story/Finance/Calendar المعتمدة سابقاً.

==================================================
6. مصدر الحقيقة
==================================================

مصدر الحقيقة لهذه المرحلة:

notifications و notification_preferences إن كانت موجودة أو ما يعادلها عبر Data Layer / service layer الحالي.

ممنوع:
- تكرار الإشعارات داخل components.
- قراءة مركز الإشعارات من مصدر وكتابة /admin إلى مصدر آخر.
- جعل العداد محسوباً من بيانات مختلفة عن القائمة.
- فقدان read/unread بعد reload إذا كان المصدر الحالي يدعم الحفظ.
- ادعاء Push أو Firebase إذا المنفذ داخلي فقط.

المطلوب:
- إشعار جديد من /admin يظهر في مركز الإشعارات.
- الإشعار الجديد يظهر كغير مقروء.
- العداد يزيد عند وصول إشعار غير مقروء.
- عند تحويله إلى مقروء، العداد ينخفض.
- الحالة تبقى بعد reload.

==================================================
7. المتطلبات التفصيلية
==================================================

أ. مركز الإشعارات:
- قائمة إشعارات واضحة.
- عنوان الإشعار.
- نص الإشعار.
- نوع الإشعار.
- تاريخ/وقت الإرسال.
- حالة مقروء/غير مقروء.
- زر تحويل إلى مقروء.
- حالة فارغة واضحة.
- حالة خطأ واضحة.

ب. لوحة المالك /admin:
- نموذج إرسال إشعار داخلي.
- حقول واضحة:
  - العنوان.
  - الرسالة.
  - النوع.
  - أولوية أو حالة إن كانت البنية تدعم ذلك.
- زر إرسال فعلي.
- feedback عند النجاح أو الفشل.
- الإشعار يظهر للمستخدم بعد الإرسال.

ج. العداد:
- يحسب عدد غير المقروء.
- يظهر في المكان الحالي المناسب إن كان موجوداً.
- إن لم يكن له مكان حالي، اعرضه في مركز الإشعارات بوضوح.
- لا تنشئ إعادة تصميم واسعة للهيدر الآن.

د. الأنواع:
استخدم الأنواع المتاحة أو ما يعادلها:
- راتب.
- دعم.
- فاتورة.
- موعد شخصي.
- صلاة.
- ستوري اليوم.
- خبر.
- وظيفة.
- رسالة من المالك.

هـ. التفضيلات:
إذا كانت notification_preferences موجودة:
- تحقق من تفعيل/إيقاف نوع إشعار.
- لا تبنِ نظاماً ضخماً إذا غير موجود.
- وثّق القيد إن بقي مؤجلاً.
