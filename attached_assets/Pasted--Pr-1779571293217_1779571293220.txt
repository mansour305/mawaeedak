تصرف كفريق بناء منتجات رقمي متكامل، وليس كمساعد يشرح أو يقترح.

أنت الآن مسؤول عن بناء تطبيق رسمي Production MVP لمنصة عربية سعودية باسم "مواعيدك"، قابل للتشغيل الحقيقي، والنشر، والتطوير لاحقاً إلى تطبيق iOS وAndroid.

الفريق الذي تمثله:
Product Manager + UI/UX Designer + Frontend Engineer + Backend Engineer + Database Architect + QA Tester + Security Reviewer + DevOps Engineer.

اسم المشروع:
Mawaeedak Production MVP v1

اسم المنصة:
مواعيدك

==================================================
0. القاعدة الحاكمة العليا
==================================================

الهدف ليس بناء Prototype.
الهدف ليس بناء واجهة تجريبية.
الهدف ليس بناء صفحات شكلية.
الهدف هو بناء تطبيق رسمي Production MVP يعمل فعلياً، قابل للنشر، وقابل للتطوير.

ممنوع استخدام مصطلح Prototype كهدف للمشروع أو داخل واجهة المستخدم.
استخدم مصطلحات:
- Production MVP
- الإصدار الرسمي الأول
- تطبيق رسمي قابل للنشر

Fallback مسموح فقط عند وجود قيد بيئي يمنع التكامل الحقيقي، ويجب عزله داخل fallback-services مع توضيح أنه مؤقت وقابل للاستبدال.

الأولوية:
1. تطبيق يعمل فعلياً.
2. Backend حقيقي أو جاهز للربط الحقيقي فور توفر المفاتيح.
3. Database حقيقية أو Schema جاهز للتنفيذ.
4. Auth حقيقي أو بنية Auth كاملة.
5. صلاحيات مستخدم ومالك.
6. أمان أساسي.
7. نشر Web/PWA.
8. قابلية التحويل إلى Expo Mobile App.
9. قابلية الوصول لاحقاً إلى App Store وGoogle Play.

أي تعارض بين الجمال البصري وكمال الوظائف يُحسم لصالح الوظائف.
أي تعارض بين كثرة الصفحات وعمق التجربة يُحسم لصالح تجربة تعمل فعلياً.
أي تعارض بين السرعة والجودة الأساسية يُحسم لصالح الجودة الأساسية.

لا تعتبر المشروع مكتملاً إلا إذا:
- الواجهات الأساسية تعمل فعلياً.
- الأزرار تنتج أثراً واضحاً.
- البيانات قابلة للإضافة والتعديل والحذف.
- لوحة المالك تؤثر فعلياً على واجهة المستخدم.
- النظام قابل للتشغيل والبناء والنشر.
- لا توجد صفحات فارغة.
- لا توجد أزرار معطلة.
- لا يوجد ادعاء تكامل غير منفذ.
- لا يوجد ادعاء بيانات رسمية غير موثقة.
- لا توجد أسرار أو مفاتيح API داخل الكود.

1. ممنوعات صارمة
==================================================

ممنوع:
- لا صفحات وهمية.
- لا أزرار بلا وظيفة.
- لا TODO ظاهر في الواجهة.
- لا coming soon.
- لا placeholder كبديل عن وظيفة مطلوبة.
- لا hardcoded data داخل صفحات المستخدم عندما يجب أن تُدار من لوحة المالك.
- لا بيانات شخصية حقيقية.
- لا أسرار أو مفاتيح API حقيقية داخل الكود.
- لا استخدام شعارات رسمية سعودية مطابقة مثل السيفين والنخلة الرسميين.
- لا ادعاء وجود تكامل خارجي حقيقي إذا لم يتم تنفيذه فعلياً.
- لا ادعاء وجود رابط معاينة أو نشر إذا لم توفره البيئة فعلياً.
- لا ادعاء نجاح اختبارات لم تُنفذ.
- لا بناء الصفحة الرئيسية فقط.
- لا ترك لوحة المالك شكلية.
- لا ترك التقويم أو المال أو ستوري اليوم أو الإشعارات بدون وظائف فعلية.
- لا استخدام كلمة "رسمي" للبيانات إلا إذا كان مصدرها الرسمي محدداً ومخزناً في بيانات العنصر.

استخدم رموزاً مستوحاة فقط من الهوية السعودية مثل:
النخلة، الجمل، الخيمة، السدو، القهوة، المجلس، الصحراء، الأقواس، الزخارف.

==================================================
2. القرار التقني الرسمي
==================================================

استخدم افتراضياً:
- Frontend: React + TypeScript + Vite.
- Styling: Tailwind CSS.
- Routing: React Router أو أفضل بديل مناسب.
- Theme Engine: CSS Variables.
- Backend/Data: Supabase.
- Database: PostgreSQL.
- Auth: Supabase Auth.
- Authorization: Supabase Row Level Security.
- Storage إن احتجنا لاحقاً: Supabase Storage.
- Notifications: Notification Center داخلي الآن، وبنية جاهزة لاحقاً لـ Firebase Cloud Messaging.
- PWA: Manifest + Service Worker قدر الإمكان.
- Mobile لاحقاً: Expo + React Native.
- Deployment: Web/PWA أولاً.
- RTL: دعم كامل للغة العربية.
- Responsive: Mobile-first.

إذا كانت مفاتيح Supabase غير متوفرة داخل البيئة:
- لا تضع مفاتيح وهمية داخل الكود.
- أنشئ ملف .env.example.
- أنشئ ملف ENV_EXAMPLE.md.
- أنشئ SUPABASE_SCHEMA.sql.
- أنشئ RLS_POLICIES.sql.
- أنشئ Supabase service layer قابلاً للتفعيل فور إضافة المفاتيح.
- استخدم local fallback مؤقت فقط حتى يعمل التطبيق داخل البيئة، مع عزله بوضوح.
- لا تجعل LocalStorage هو الحل الرئيسي للبيانات الرسمية.
- لا تدّع أن Supabase متصل فعلياً إذا لم تكن المفاتيح مضافة ولم يتم الربط.

LocalStorage يستخدم فقط لـ:
- حفظ الثيم.
- حفظ تفضيلات الواجهة.
- fallback مؤقت عند غياب Supabase.
- بيانات تجربة محلية أثناء التطوير.
- حفظ حالة واجهة لا تُعد بيانات رسمية.

==================================================
4. تعريف “يعمل فعلياً”
==================================================

أي وظيفة لا تنتج أثراً واضحاً تعتبر غير منفذة.

الأثر المقبول:
- فتح صفحة حقيقية.
- فتح نافذة إدخال.
- حفظ بيانات.
- تعديل بيانات.
- حذف بيانات.
- تغيير حالة.
- ظهور البيانات في شاشة أخرى.
- تطبيق ثيم على كامل التطبيق.
- إنشاء إشعار داخلي.
- إنشاء ستوري قابل للمعاينة.
- حساب نتيجة فعلية.
- فلترة أو بحث فعلي.
- حفظ التغيير بعد إعادة تحميل الصفحة قدر الإمكان.
- تحديث البيانات من لوحة المالك وظهورها للمستخدم.
- تسجيل العملية في audit_logs عند كونها عملية إدارية.

زر لا يفعل شيئاً = فشل.
صفحة فارغة = فشل.
بطاقة لا تفتح شيئاً عند الحاجة = فشل.
بيانات لا يمكن تعديلها وهي مطلوبة من لوحة المالك = فشل.
ادعاء تكامل غير منفذ = فشل.

9. شاشة البداية والترحيب
==================================================

ابنِ شاشة بداية تحتوي على:
- اسم "مواعيدك".
- عبارة:
  مواعيدك — كل موعد له وقته
- مؤثر تحميل بسيط وفخم.
- تطبيق الثيم الرسمي.

ابنِ شاشة ترحيب أول استخدام تحتوي على:
1. تعريف مختصر بالمنصة.
2. اختيار المدينة.
3. طلب تفعيل الإشعارات بالنص:
   فعّل الإشعارات حتى نذكّرك برواتبك، دعمك، صلاتك، وفواتيرك ومواعيدك المهمة في وقتها.
4. اختيار الاهتمامات:
   الرواتب، حساب المواطن، الضمان، الدعم السكني، التقاعد، التأمينات، حافز، ساند، التأهيل، الفواتير، الصلاة، الدراسة، الإجازات، الأخبار، الوظائف.
5. اختيار الثيم، مع جعل "التراث التقني الفاخر" مفعلاً افتراضياً.

إذا كان Supabase Auth غير مفعّل بسبب عدم توفر المفاتيح:
- ابنِ واجهة تسجيل دخول حقيقية.
- ابنِ auth service قابل للربط.
- استخدم demo mode مؤقت فقط داخل التطوير.
- لا تعرض demo mode كأنه النظام النهائي.
- وضّح في التسليم أن تفعيل تسجيل الدخول الحقيقي يتطلب مفاتيح Supabase.

10. الرئيسية
==================================================

ابنِ الصفحة الرئيسية كأقوى شاشة في التطبيق.

العناصر المطلوبة:
- شريط علوي يحتوي على:
  الرئيسية، زر القائمة، زر مشاركة، زر الرسائل، زر إخفاء الإعلانات، زر الإشعارات.
- بطاقة تاريخ اليوم:
  اليوم الحالي، التاريخ الهجري، التاريخ الميلادي.
- بطاقة رسالة اليوم من daily_messages.
- بطاقة مواقيت الصلاة:
  الفجر، الشروق، الظهر، العصر، المغرب، العشاء، الصلاة القادمة، الوقت المتبقي، المدينة الحالية.
- بطاقة "كم تبقى على":
  الراتب، حساب المواطن، الضمان، حافز، الدعم السكني، التقاعد، التأهيل، ساند/التأمينات، الدعم الزراعي.
- قائمة المواعيد القادمة.
- زر إنشاء ستوري اليوم.
- زر إضافة موعد سريع.
- اختصارات سريعة للأقسام.
- عداد إشعارات غير مقروءة.

العدادات يجب أن تُحسب من التاريخ الحالي.

مواقيت الصلاة:
- إذا توفر تكامل أو مكتبة موثوقة، استخدمها.
- إذا لم يتوفر، استخدم Seed Data قابلة للتعديل من لوحة المالك.
- احسب الصلاة القادمة والوقت المتبقي بناءً على البيانات المتاحة.
- لا تدّع أنها أوقات رسمية دقيقة إذا كانت Seed أو fallback.

التاريخ الهجري:
- استخدم Intl.DateTimeFormat مع islamic-umalqura إن كان مدعوماً.
- إذا لم يكن مدعوماً، استخدم fallback واضح.

زر إخفاء الإعلانات:
- في الإصدار الأول، لا تبنِ نظام إعلانات حقيقي إلا إذا طُلب صراحة.
- اجعل زر إخفاء الإعلانات تفضيلاً واجهياً محفوظاً في user preferences.
- لا تربطه بدفع أو اشتراك.
- لا تدّع وجود نظام إعلانات فعلي.

==================================================
18. قاعدة البيانات و Supabase
==================================================

صمّم النظام حول قاعدة بيانات حقيقية قابلة للتنفيذ في Supabase/PostgreSQL.

الجداول المطلوبة:

- profiles
- admin_users
- roles
- permissions
- role_permissions
- user_roles
- financial_events
- public_events
- appointments
- appointment_categories
- prayer_settings
- daily_messages
- story_templates
- themes
- notifications
- notification_preferences
- news
- jobs
- complaints
- audit_logs
- app_settings

لا تنشئ جدول public.users إذا كان سيتعارض مع auth.users.
استخدم profiles كجدول المستخدمين العام المرتبط بـ auth.users.id.
profiles.id يفضّل أن يكون UUID مطابقاً لـ auth.users.id عند استخدام Supabase Auth.
admin_users يجب أن يرتبط بـ profiles.id أو auth.users.id، ولا يكون جدولاً منفصلاً بلا علاقة بالمستخدم الحقيقي.
user_roles يجب أن يربط profiles.id أو auth.users.id مع roles.id.
أي بيانات مستخدم يجب أن ترتبط بـ auth.users.id أو profiles.id بوضوح.

أنشئ:
- SUPABASE_SCHEMA.sql
- RLS_POLICIES.sql

RLS يجب أن يراعي:
- المستخدم يرى بياناته فقط.
- المالك يرى بيانات الإدارة.
- العمليات الحساسة للمالك فقط.
- الأخبار والمواعيد العامة قابلة للقراءة حسب الحالة.
- حذف وتعديل السجلات الإدارية يحتاج صلاحية.
- audit_logs للقراءة الإدارية فقط.

إذا لم تكن مفاتيح Supabase متوفرة:
- ابنِ service layer مطابق للجداول.
- استخدم fallback مؤقت.
- اجعل النقل إلى Supabase لاحقاً لا يتطلب إعادة بناء الواجهة.

- ترسل من لوحة المالك.
- تظهر في مركز الإشعارات.
- تظهر كغير مقروءة.

- يمكن تحويلها إلى مقروءة.

- تُدار من لوحة المالك.
- تظهر في المراكز.
- يمكن البحث أو الفلترة.

- يأخذ التاريخ ورسالة اليوم والعدادات من Data Layer.
- يسمح بتعديل النص.
- يسمح باختيار العناصر.

- يسمح بالحفظ أو النسخ أو المشاركة.

==================================================

أي ميزة يتعذر تنفيذها بالكامل بسبب قيود البيئة يجب ألا تُترك فارغة.

إذا تعذر التكامل الخارجي:

- ابنِ تدفق المستخدم.
- اكتب بوضوح أنها جاهزة للتفعيل عند إضافة المفاتيح.

إذا تعذر الحفظ الحقيقي:
- استخدم fallback مؤقت.
- لا تجعله التصميم النهائي.

إذا تعذر توليد صورة ستوري:
- وفر معاينة تصميمية.
- وفر نسخ النص.
- وفر حفظ داخلي.

إذا تعذر إرسال Push Notification:
- وفر إشعاراً داخلياً يظهر في Notification Center.

إذا تعذر استخدام API للصلاة:
- استخدم بيانات قابلة للإدارة.
- احسب الصلاة القادمة من البيانات المتاحة.

ممنوع ترك أي ميزة مطلوبة فارغة بحجة عدم توفر التكامل.

27. أوضاع التنفيذ حسب توفر Supabase
==================================================

قبل البناء، افحص هل مفاتيح Supabase متوفرة في البيئة.

Mode A — Supabase Available:
إذا كانت مفاتيح Supabase متوفرة:
- اربط Supabase فعلياً.
- استخدم Supabase Auth.
- استخدم Supabase client.
- اجعل البيانات الأساسية تُقرأ وتُكتب من Supabase.
- طبّق RLS فعلياً واختبره.
- لا تستخدم fallback إلا عند فشل الاتصال.
- لا تضع أسراراً داخل الكود.
- استخدم متغيرات البيئة فقط.

Mode B — Supabase Not Available:
إذا لم تكن مفاتيح Supabase متوفرة:
- لا تدّع أن backend متصل فعلياً.
- ابنِ Supabase service layer مطابقاً للجداول.
- ابنِ fallback local service مؤقت.
- أنشئ SUPABASE_SCHEMA.sql و RLS_POLICIES.sql.
- اجعل كل واجهة تستخدم service interface موحداً بحيث يمكن استبدال fallback بـ Supabase دون إعادة بناء الواجهة.
- اذكر في التسليم أن تفعيل البيانات الرسمية يتطلب إضافة مفاتيح Supabase وتشغيل SQL.

==================================================
28. أدوار المستخدمين والصلاحيات

أنشئ الأدوار التالية:

4. content_manager

29. سياسة مصادر البيانات
==================================================

أي بيانات مالية أو حكومية أو دعم أو رواتب أو إجازات لا تُعرض على أنها رسمية إلا إذا كانت مرتبطة بمصدر رسمي واضح.

في الإصدار الأول:
- مواعيد الرواتب والدعم تُدار من لوحة المالك.
- الأخبار والوظائف تُدار من لوحة المالك.
- مواقيت الصلاة إن لم تكن من مكتبة أو API موثوق تعرض كبيانات قابلة للتعديل.

يجب وضع الحقول التالية في السجلات المهمة:
- source_type: official | admin_managed | estimated | fallback
- source_name
- last_verified_at
- status

في واجهة المستخدم:
لا تكتب "رسمي" إلا إذا كان source_type = official.

30. الصفحات القانونية الأساسية

لأن التطبيق رسمي، أنشئ الصفحات التالية:

- /terms : شروط الاستخدام
- /disclaimer : إخلاء مسؤولية البيانات
- /support : الدعم والمساعدة

يجب أن تكون النصوص عربية واضحة، وتوضح:

- أن المستخدم مسؤول عن التحقق من المواعيد الرسمية عند الحاجة.
- أن الحاسبات تقديرية ما لم تكن مرتبطة بمصدر رسمي.
- أن المستخدم يستطيع طلب حذف حسابه.
- أن التطبيق لا يخزن مفاتيح أو بيانات حساسة داخل الواجهة.

النصوص القانونية في الإصدار الأول تكون مسودة تشغيلية، ويجب توثيق أنها تحتاج مراجعة قانونية قبل الإطلاق التجاري الكامل.

أضف روابط هذه الصفحات داخل صفحة حسابي وFooter الويب.

- اجعل مشاركة النصوص والنسخ والإشعارات تمر عبر abstraction layer.
- وثّق في ARCHITECTURE.md ما يمكن نقله إلى Expo وما يحتاج إعادة بناء.

==================================================

حاسبة نهاية الخدمة والراتب الصافي يجب أن تكون تقديرية في الإصدار الأول ما لم يتم ربطها بمصدر رسمي واضح.

لا تدّع أن نتيجة الحاسبة رسمية أو قانونية.

36. ضبط وظائف التقديم

- إذا لم يوجد رابط، افتح صفحة تفاصيل الوظيفة فقط.
- لا تدّع وجود تقديم رسمي إذا لم يكن الرابط متوفراً.

==================================================
37. بوابة الجاهزية الرسمية Release Gate
==================================================

فرّق بوضوح بين حالتين:

1. Publishable Preview
تعني:
- التطبيق يعمل داخل بيئة التطوير أو رابط معاينة.
- بعض الخدمات قد تعمل عبر fallback مؤقت.
- Supabase أو Firebase قد لا تكون مفعلة.
- لا يجوز تسميته Production Ready بالكامل.

2. Production Ready
تعني:
- Supabase متصل فعلياً.
- Auth يعمل فعلياً.
- RLS مفعّل ومختبر فعلياً.
- لا توجد بيانات رسمية مزعومة بلا مصدر.
- لا توجد أسرار داخل الكود.
- build ناجح.
- الصفحات القانونية موجودة.
- لوحة المالك محمية بالصلاحيات.
- التوثيق مكتمل.

ممنوع استخدام عبارة "جاهز إنتاجياً" أو "Production Ready" إذا كان النظام يعمل فقط عبر fallback محلي.
في هذه الحالة استخدم:
"Publishable Preview جاهز للتجربة، ويتطلب تفعيل Supabase للوصول إلى Production Ready."

==================================================
38. حماية Supabase والمفاتيح
==================================================

ممنوع نهائياً:
- وضع service_role key داخل الواجهة.
- وضع مفاتيح سرية داخل ملفات عامة.
- استخدام أي مفتاح إداري في client-side code.
- تجاوز RLS من الواجهة.

المسموح:
- استخدام anon public key فقط في الواجهة عند الحاجة.
- حفظ المفاتيح في environment variables.
- تنفيذ العمليات الإدارية الحساسة عبر سياسات RLS أو طبقة آمنة لاحقاً.

إذا احتاجت عملية إدارية صلاحية عالية:
- لا تنفذها بمفتاح service_role في المتصفح.
- وثّق أنها تحتاج Edge Function أو backend secure endpoint لاحقاً.
- لا تدّع أنها آمنة إنتاجياً إذا كانت تعمل محلياً فقط.

41. الوصول وسهولة الاستخدام Accessibility
==================================================

يجب مراعاة:
- تباين ألوان واضح.
- أزرار مناسبة للمس على الجوال.
- aria-label للأيقونات والأزرار التي بلا نص.
- focus states للكيبورد.
- أحجام خط مقروءة.
- عدم الاعتماد على اللون وحده لتمييز الحالة.
- دعم RTL في كل الحقول والقوائم والنوافذ.

أي زر أيقوني مثل:
الإشعارات، المشاركة، القائمة، الرسائل، إخفاء الإعلانات
يجب أن يحتوي label واضحاً للمستخدم وaria-label في الكود.

- background_color.
- icons placeholders آمنة أو مولّدة داخل المشروع.

- offline fallback بسيط.
- installability basics.

إذا تعذر تفعيل service worker:
- وثّق السبب في QA_REPORT.md.
- لا تدّع أن PWA مكتمل إذا كان manifest فقط.

43. جودة الأداء
==================================================

- قلل التكرار في المكونات.
- لا تضع بيانات ضخمة داخل component render.
- افصل الحسابات الثقيلة في helpers.
- تجنب إعادة الحساب غير الضرورية للعدادات.

إذا كانت البيئة تدعم Lighthouse أو فحص مشابه، شغله أو وثّق عدم توفره.

46. حذف الحساب والبيانات
==================================================

إذا Supabase متصل:

- لا تحذف بيانات عامة أو إدارية.
- سجّل العملية في audit_logs إن كانت إدارية.

إذا Supabase غير متصل:
- نفّذ حذفاً محلياً لبيانات fallback الخاصة بالمستخدم.
- وضّح أن هذا حذف محلي داخل بيئة التطوير فقط.
- لا تدّع أنه حذف حساب حقيقي من الخادم.

51. مراجعة الواجهة النهائية
==================================================

- هل يعرف المستخدم ماذا يفعل؟
- هل التبويبات مفهومة؟
- هل النصوص العربية طبيعية؟
- هل توجد كلمات إنجليزية غير ضرورية؟
- هل البطاقات مزدحمة؟

- هل يوجد فرق واضح بين البيانات الرسمية والتقديرية؟
- هل لوحة المالك منفصلة بصرياً عن تطبيق المستخدم؟
- هل الثيم الفاخر واضح بدون مبالغة؟
- هل التطبيق يبدو سعودياً حديثاً دون استخدام شعار رسمي ممنوع؟

أي خلل بصري واضح يجب إصلاحه قبل التسليم.

==================================================
52. مراجعة عدم التناقض
==================================================

قبل التسليم، افحص برمجياً ووظيفياً عدم وجود تناقضات مثل:

- صفحة تقول إن Supabase متصل بينما هو fallback.
- زر يبدو مدفوعاً ولا يوجد دفع.
- بيانات تقول رسمية وهي admin_managed.
- إشعارات Push بينما المنفذ داخلي فقط.
- عضوية أو ميزة مدفوعة غير منفذة.
- رابط تقديم وظيفة غير موجود.
- حاسبة قانونية بلا تنبيه تقديري.
- /admin ظاهر للمستخدم العادي.
- حذف حساب بلا تأكيد.
- ثيم مطبق جزئياً فقط.
- Story يأخذ نصاً ثابتاً بدلاً من Data Layer.

أي تناقض من هذه القائمة يمنع التسليم.

قبل التسليم، يجب أن تكون ملفات التوثيق محدثة مع الواقع الفعلي، لا مع الخطة.

README.md يجب أن يطابق طريقة التشغيل الفعلية.
ARCHITECTURE.md يجب أن يشرح البنية المنفذة فعلاً.
QA_REPORT.md يجب أن يذكر الاختبارات المنفذة فعلاً.
FALLBACK_SERVICES.md يجب أن يوضح أي fallback فعلي.
ENV_EXAMPLE.md يجب أن يحتوي المتغيرات المطلوبة فعلاً.
SUPABASE_SCHEMA.sql يجب أن يطابق الجداول المستخدمة في services.
RLS_POLICIES.sql يجب أن يطابق الأدوار والصلاحيات المذكورة.

ممنوع كتابة توثيق يدّعي شيئاً لم ينفذ.

==================================================
54. مصدر الحقيقة الواحد Single Source of Truth
==================================================

يجب أن يكون لكل نوع بيانات مصدر حقيقة واحد واضح.

ممنوع تكرار نفس البيانات في:
- components
- seed data
- fallback service
- admin state
- UI constants

إلا إذا كانت نسخة عرض مؤقتة موثقة.

القاعدة:
- واجهة المستخدم تقرأ من service layer.
- لوحة المالك تكتب إلى service layer.
- service layer يقرر هل المصدر Supabase أو fallback.
- المكونات لا تعرف هل البيانات من Supabase أو fallback.
- أي تعديل إداري يجب أن يمر من نفس مصدر البيانات الذي تقرأ منه واجهة المستخدم.

إذا اكتشفت بيانات مكررة في أكثر من مكان:
- انقلها إلى data/service layer.
- اجعل المكونات تستدعي hooks/services فقط.

==================================================
55. حماية الإدارة ليست إخفاء واجهة فقط
==================================================

إخفاء رابط /admin من تبويبات المستخدم لا يكفي.

يجب تنفيذ حماية منطقية:
- route guard لصفحات /admin.
- role check قبل عرض أي صفحة إدارية.
- منع المستخدم العادي من تنفيذ عمليات الإدارة.
- عند استخدام fallback، طبّق نفس منطق الصلاحيات داخلياً.
- عند استخدام Supabase، يجب أن تعتمد الحماية النهائية على RLS وليس على الواجهة فقط.

ممنوع اعتبار لوحة المالك محمية إذا كان يمكن فتح /admin كمستخدم عادي بدون صلاحية.

==================================================
56. قواعد قاعدة البيانات الرسمية
==================================================

في SUPABASE_SCHEMA.sql يجب مراعاة الآتي قدر الإمكان:

- استخدام UUID كمفتاح أساسي للجداول المهمة.
- created_at و updated_at في الجداول الرئيسية.
- created_by و updated_by في الجداول الإدارية.
- user_id أو owner_id لأي بيانات تخص مستخدماً محدداً.
- is_active أو status للجداول التي تظهر في الواجهة العامة.
- فهارس Indexes للحقول المستخدمة في البحث والفلترة.
- علاقات واضحة بين roles و permissions و role_permissions و user_roles.
- عدم الاعتماد على email وحده كمفتاح علاقة.

أي جدول يحتوي بيانات مستخدم يجب أن يكون قابلاً لتطبيق RLS عليه بوضوح.

63. شروط منع التسليم
==================================================

ممنوع إعلان المشروع مكتمل إذا تحقق أي بند:

- توجد صفحة فارغة.
- توجد بطاقة لا تفتح شيئاً وهي يفترض أن تفتح.
- التقويم لا يدعم إضافة/تعديل/حذف.
- المواعيد لا تظهر في الرئيسية.
- المال لا يحتوي عدادات فعلية.
- الحاسبات لا تنتج نتائج.
- الحاسبات النظامية لا تعرض أنها تقديرية.
- ستوري اليوم لا يعرض معاينة.
- الثيم لا يغير كامل النمط البصري.
- الإشعارات لا تظهر في مركز الإشعارات.
- لوحة المالك لا تغير بيانات تظهر للمستخدم.
- البيانات تضيع بعد إعادة التحميل رغم إمكانية حفظها.
- الواجهة ليست RTL بالكامل.
- يوجد TODO أو placeholder ظاهر للمستخدم.
- يوجد خطأ build.
- يوجد ادعاء تكامل خارجي غير منفذ.
- يوجد ادعاء بيانات رسمية وهي غير رسمية.
- لا توجد ملفات توثيق.
- لا يوجد Schema أو RLS للربط الرسمي.
- لا توجد صفحات قانونية.
- لا يوجد تمييز بين Supabase الفعلي وfallback المؤقت.
- لا توجد أدوار وصلاحيات واضحة.
- لا توجد سياسة مصادر بيانات داخل data model.
- لا توجد صفحة 404.
- لا يوجد Error Boundary.
- توجد شاشة بيضاء عند الخطأ.
- يوجد service_role key أو مفتاح سري داخل الواجهة.
- يستخدم التطبيق "Production Ready" وهو يعمل فقط عبر fallback.
- توجد عضوية أو اشتراك أو دفع غير منفذ.
- توجد إشعارات Push مزعومة بينما المنفذ داخلي فقط.
- التوثيق لا يطابق الواقع الفعلي.
- تتكرر البيانات الأساسية في أكثر من مصدر حقيقة.
- حماية /admin تعتمد على إخفاء الواجهة فقط.
- الجداول لا تحتوي ownership واضحاً للبيانات الخاصة بالمستخدم.
- توجد مكتبات كثيرة غير مبررة.
- توجد مصطلحات عربية وإنجليزية مختلطة بلا ضرورة.

67. تنظيفات إنتاجية نهائية
==================================================

استبدل أي دلالة قد توحي بالدفع أو الاشتراك في الإصدار الأول بما يلي:
- مجاني
- مميز داخلي
- غير متاح حالياً
- للمالك فقط

ممنوع استخدام VIP داخل واجهة المستخدم إذا كان قد يوحي باشتراك أو دفع غير منفذ.

في Notification Center:
استخدم عبارة "إشعار داخلي" بدلاً من "إشعار تجريبي" داخل واجهة المستخدم الرسمية.
يمكن استخدام كلمة "تجريبي" فقط داخل QA_REPORT.md أو وثائق التطوير.

في Supabase:
لا تنشئ جدول public.users إذا كان سيتعارض مع auth.users.
استخدم profiles كجدول المستخدمين العام المرتبط بـ auth.users.id.
أي بيانات مستخدم يجب أن ترتبط بـ auth.users.id أو profiles.id بوضوح.

قبل التسليم:
راجع كل نص ظاهر للمستخدم وتأكد أنه لا يحتوي:
- تجريبي
- مؤقت
- fallback
- mock
- VIP
إلا إذا كان السياق يشرح بوضوح أنه داخل بيئة تطوير أو وثائق تقنية.

يسمح باستخدام fallback داخل أسماء الملفات والتوثيق التقني فقط، وليس داخل واجهة المستخدم.
