استمر من آخر Checkpoint. لا تعيد البناء من البداية.

المرحلة الحالية:
Phase 12N
Calendar Appointments Gateway Cutover Gate

الدور:
تصرف كفريق Data Layer Architect + Supabase Engineer + Frontend Reliability Engineer + QA Lead + Security Reviewer.
مهمتك الآن ليست تحويل كل النظام إلى Supabase، وليست حذف API، وليست تحويل Finance.
مهمتك فقط تحويل قراءة وكتابة المواعيد appointments إلى Gateway بشكل آمن ومتسق، مع الحفاظ على API كـ fallback وعدم كسر HomePage أو CalendarPage.

==================================================
0. السياق الحالي
==================================================

تم اعتماد Phase 12M:
- Admin Themes أصبح Gateway.
- Admin Story Templates أصبح Gateway.
- Admin Daily Messages أصبح Gateway.
- Admin News/Jobs سابقاً أصبح Gateway.
- Notification System أصبح Gateway.
- CalendarPage و FinancePage لا يزالان على Orval/API.
- API وPostgreSQL ما زالا موجودين.
- default data source = api.
- mode=api يعمل.
- mode=supabase_shadow يعمل.
- mode=supabase يعمل للنطاقات المحولة.
- build ناجح.
- typecheck ناجح.
- لا service_role.
- لا أسرار hardcoded.
- لا DROP/TRUNCATE/DELETE غير مقصود.

حالة بيانات Supabase:
- appointments = 2
- appointments مرتبطة بـ user_id الخاص بـ hrq@hotmail.com.
- legacy_id محفوظ.
- migration_batch محفوظ.

القيد الحالي:
- CalendarPage CRUD ما زال Orval/API.
- HomePage upcoming appointments غالباً ما زال يقرأ من API.
- عند mode=supabase، أي صفحة تقرأ appointments من Supabase بينما تكتب API قد تسبب divergence.
- لذلك هذه المرحلة يجب أن تجعل قراءة وكتابة appointments متسقة عبر Gateway.

==================================================
1. الهدف التنفيذي
==================================================

حوّل appointments فقط إلى Gateway:

1. CalendarPage list/read appointments.
2. CalendarPage add appointment.
3. CalendarPage edit appointment.
4. CalendarPage delete appointment.
5. HomePage upcoming appointments read إن كانت تستخدم appointments.
6. أي hook مرتبط بالمواعيد القادمة.

المطلوب حسب mode:

أ. mode=api:
- القراءة والكتابة عبر API كما كانت.
- لا تغيير سلوك المستخدم.

ب. mode=supabase_shadow:
- القراءة والكتابة عبر API.
- لا كتابة إلى Supabase.
- يمكن إجراء shadow comparison فقط.

ج. mode=supabase:
- قراءة appointments من Supabase.
- إضافة appointment إلى Supabase.
- تعديل appointment في Supabase.
- حذف appointment من Supabase.
- HomePage upcoming يقرأ من نفس مصدر CalendarPage قدر الإمكان.
- لا fallback صامت إلى API في write.
- عند فشل write، أظهر toast/error واضح.

ممنوع تحويل:
- financial_events.
- notifications.
- news/jobs/themes/story/daily_messages لأنها منتهية ولا تُلمس إلا إذا لزم invalidation.
- profiles.
- audit_logs.

4. افحص dataGateway.ts.

7. افحص هل UI يستخدم id كـ legacy integer أو uuid.
8. افحص fields:

   - date
   - time
   - type/category إن وجد
   - notes/description إن وجد
   - user_id
   - legacy_id
   - is_active/status إن وجد

9. حدد mapping واضح:
   - id في الواجهة = legacy_id إن كان legacy موجوداً.
   - عند إنشاء صف جديد في Supabase، ما قيمة legacy_id؟
   - إذا legacy_id مطلوب لتوافق الواجهة، ضع استراتيجية آمنة:
     أ. إما توليد legacy_id جديد من max(legacy_id)+1 داخل Supabase.
     ب. أو استخدام uuid وإرجاع id متوافق موثق.
   - لا تنفذ شيئاً حتى توثق القرار.

10. افحص RLS:
   - المستخدم يرى مواعيده فقط.
   - المستخدم يضيف مواعيده فقط.
   - المستخدم يعدّل مواعيده فقط.
   - المستخدم يحذف مواعيده فقط.

11. حدد استراتيجية invalidation بعد كل mutation:
   - Calendar list.
   - HomePage upcoming.
   - أي Orval keys متبقية للتوافق.

- gwUpdateAppointment.
- gwDeleteAppointment.

أضف أو حدّث hooks في useGatewayData أو المكان المناسب:

- useGatewayAppointments.
- useGatewayUpcomingAppointments.
- mutations للـ create/update/delete إن كان النمط موجوداً.

د. CalendarPage:
- القراءة عبر Gateway.
- create/edit/delete عبر Gateway.
- لا تكسر ConfirmDialog.
- لا تكسر toast.
- لا تكسر reload behavior.

هـ. HomePage:
- upcoming appointments عبر Gateway إذا كان آمناً.
- يجب أن يرى نفس بيانات CalendarPage في mode=supabase.

و. السلوك حسب mode:
- api: API.
- supabase_shadow: API.
- supabase: Supabase.

ز. Write behavior:
- لا fallback صامت إلى API عند mode=supabase.
- إذا فشل Supabase write، اعرض خطأ واضح.
- لا تكتب API وSupabase معاً.
