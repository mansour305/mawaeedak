تصرف كفريق بناء منتجات رقمي متكامل، وليس كمساعد يشرح أو يقترح.

أنت الآن مسؤول عن بناء نسخة AI Prototype فعلية وعاملة لمنصة عربية سعودية باسم "مواعيدك"، وليس كتابة خطة أو ملخص أو وصف نظري.

الفريق الذي تمثله:
Product Manager + UI/UX Designer + Frontend Engineer + Backend Engineer + Database Architect + QA Tester + Security Reviewer.

اسم المشروع:
Mawaeedak AI Prototype v1

اسم المنصة:
مواعيدك

==================================================
القاعدة الحاكمة العليا
==================================================

أنت لا تكتب وصفاً ولا خطة ولا اقتراحات.
أنت تنفذ Prototype يعمل فعلياً داخل بيئة التطوير المتاحة.

أي تعارض بين الجمال البصري وكمال الوظائف يُحسم لصالح الوظائف.
أي تعارض بين كثرة الصفحات وعمق التجربة يُحسم لصالح تجربة تعمل فعلياً.
أي تعارض بين السرعة والجودة الأساسية يُحسم لصالح الجودة الأساسية.

لا تعتبر المشروع مكتملاً إلا إذا كانت الواجهات الأساسية تعمل فعلياً، والأزرار تنتج أثراً واضحاً، والبيانات قابلة للإضافة والتعديل والحذف، وتعديلات لوحة المالك تظهر في واجهة المستخدم.

ممنوع:
- لا صفحات وهمية.
- لا أزرار بلا وظيفة.
- لا TODO داخل الواجهة.
- لا coming soon.
- لا placeholder كبديل عن وظيفة مطلوبة.
- لا hardcoded data داخل صفحات المستخدم عندما يجب أن تُدار من لوحة المالك.
- لا ملخص قبل تنفيذ البناء.
- لا الاكتفاء بالصفحة الرئيسية.
- لا اعتبار المشروع جاهزاً إذا كان التقويم أو المال أو ستوري اليوم أو لوحة المالك غير مكتمل.
- لا استخدام شعارات رسمية سعودية مطابقة مثل السيفين والنخلة الرسميين.
- لا ادعاء وجود تكامل خارجي حقيقي إذا لم يتم تنفيذه فعلياً.
- لا ادعاء وجود رابط معاينة أو نشر إذا لم توفره البيئة فعلياً.
- لا ادعاء نجاح اختبارات لم تُنفذ.

استخدم رموزاً مستوحاة فقط من الهوية السعودية مثل:
النخلة، الجمل، الخيمة، السدو، القهوة، المجلس، الصحراء، الأقواس، الزخارف.

==================================================
القرار التقني الإلزامي
==================================================

استخدم افتراضياً:
- React + TypeScript + Vite.
- Tailwind CSS.
- CSS Variables للثيمات.
- LocalStorage أو Store داخلي للحفظ التجريبي.
- Mock Services قابلة للاستبدال لاحقاً بـ Supabase وFirebase.
- PWA Manifest قدر الإمكان.
- RTL كامل.
- Mobile-first.

إذا كانت البيئة تفرض تقنية أخرى، استخدم أفضل بديل متاح بشرط ألا يقل عن نفس مستوى التنظيم.

بنية المشروع المطلوبة:
- /components
- /features
- /data
- /lib
- /hooks
- /types
- /styles
- /services أو /mock-services
- /admin

يجب فصل:
- البيانات عن الواجهة.
- الحسابات عن المكونات.
- الثيمات عن الصفحات.
- منطق الإشعارات عن العرض.
- منطق التقويم عن شكل التقويم.
- Mock services عن واجهات المستخدم.

أي كود مكرر كثيراً يجب تحويله إلى مكون أو دالة مساعدة.

==================================================
تعريف “يعمل فعلياً”
==================================================

أي وظيفة لا تنتج أثراً واضحاً تعتبر غير منفذة.

الأثر المقبول هو واحد أو أكثر من التالي:
- فتح صفحة حقيقية.
- فتح نافذة إدخال.
- حفظ بيانات.
- تعديل بيانات.
- حذف بيانات.
- تغيير حالة.
- ظهور البيانات في شاشة أخرى.
- تطبيق ثيم على كامل التطبيق.
- إنشاء إشعار داخلي.
- إنشاء ستوري قابل للمعاينة.
- حساب نتيجة فعلية.
- فلترة أو بحث فعلي.
- حفظ التغيير بعد إعادة تحميل الصفحة قدر الإمكان.

زر لا يفعل شيئاً = فشل.
صفحة فارغة = فشل.
بطاقة لا تفتح شيئاً عند الحاجة = فشل.
بيانات لا يمكن تعديلها وهي مطلوبة من لوحة المالك = فشل.

==================================================
الفكرة
==================================================

منصة سعودية ذكية تجمع مواعيد المستخدم اليومية والمالية في مكان واحد:
التاريخ الهجري والميلادي، مواقيت الصلاة، الرواتب، الدعم، الفواتير، التقويم، المواعيد الشخصية، ستوري اليوم، ومركز المالك.

الهدف التنفيذي:
بناء نسخة Web App / PWA احترافية قابلة للتجربة، تعمل على الجوال والويب، وتحتوي على الهيكل الأساسي الحقيقي للمنصة، مع واجهة عربية RTL، بيانات تجريبية منظمة، لوحة مالك مصغرة، Theme Engine، Notification Center، Data Layer داخلي، وثيم رسمي فاخر.

القرار:
ابدأ كـ Web App / PWA وليس كتطبيق متجر مباشر.
السبب: الويب أسرع في البناء والتعديل والاختبار، وبعد نجاح التجربة يمكن تحويل المنتج إلى Expo Mobile App ثم رفعه لاحقاً إلى App Store وGoogle Play.

==================================================
حدود النسخة التجريبية
==================================================

إذا تعذر استخدام API أو تكامل رسمي:
- ابنِ Mock Service واضح.
- ابنِ بنية البيانات.
- ابنِ واجهة الإدارة.
- اجعل البيانات قابلة للتعديل.
- لا تعرض البيانات كأنها رسمية مؤكدة.

ينطبق ذلك على:
- مواقيت الصلاة.
- التاريخ الهجري إن لم يكن محسوباً فعلياً.
- مواعيد الدعم.
- الأخبار.
- الوظائف.
- الحاسبات المرتبطة بأنظمة رسمية.
- الإشعارات الخارجية.
- Supabase.
- Firebase.

النص داخل الواجهة عند الحاجة:
"بيانات تجريبية قابلة للتعديل في نسخة Prototype."

إذا كانت الميزة بنية جاهزة فقط، اكتب بوضوح:
"جاهزة للربط لاحقاً، وليست تكاملاً خارجياً فعلياً."

==================================================
1. شاشة البداية والترحيب
==================================================

ابنِ شاشة بداية تحتوي على:
- شعار أو اسم "مواعيدك".
- عبارة:
  مواعيدك — كل موعد له وقته
- مؤثر تحميل بسيط وفخم.
- تطبيق الثيم الرسمي.

ابنِ شاشة ترحيب أول استخدام تحتوي على:
1. تعريف مختصر بالمنصة.
2. اختيار المدينة.
3. طلب تفعيل الإشعارات بالنص:
   فعّل الإشعارات حتى نذكّرك برواتبك، دعمك، صلاتك، وفواتيرك ومواعيدك المهمة في وقتها.
4. اختيار الاهتمامات:
   الرواتب، حساب المواطن، الضمان، الدعم السكني، التقاعد، التأمينات، حافز، ساند، التأهيل، الفواتير، الصلاة، الدراسة، الإجازات، الأخبار، الوظائف.
5. اختيار الثيم، مع جعل "التراث التقني الفاخر" مفعلاً افتراضياً.

إذا كان تسجيل الدخول غير جاهز في النسخة الأولى، استخدم مستخدماً تجريبياً مع بنية جاهزة للتسجيل لاحقاً.

==================================================
2. الرئيسية
==================================================

ابنِ الصفحة الرئيسية كأقوى شاشة في التطبيق.

العناصر المطلوبة:
- شريط علوي يحتوي على:
  الرئيسية، زر القائمة، زر مشاركة، زر الرسائل، زر إخفاء الإعلانات، زر الإشعارات.
- بطاقة تاريخ اليوم:
  اليوم الحالي، التاريخ الهجري، التاريخ الميلادي.
- بطاقة رسالة اليوم:
  رسالة يومية غير مكررة من daily_messages.
- بطاقة مواقيت الصلاة:
  الفجر، الشروق، الظهر، العصر، المغرب، العشاء، الصلاة القادمة، الوقت المتبقي للصلاة القادمة، المدينة الحالية.
- بطاقة "كم تبقى على":
  الراتب، حساب المواطن، الضمان، حافز، الدعم السكني، التقاعد، التأهيل، ساند/التأمينات، الدعم الزراعي.
- قائمة المواعيد القادمة.
- زر إنشاء ستوري اليوم.
- زر إضافة موعد سريع.
- اختصارات سريعة للأقسام.

العدادات يجب أن تُحسب من التاريخ الحالي.
إذا لم تتوفر بيانات رسمية حقيقية، استخدم Seed Data قابلة للتعديل من لوحة المالك.

مواقيت الصلاة:
إذا لم يتوفر API فعلي:
- استخدم Seed Data حسب المدينة.
- اجعلها قابلة للتعديل من لوحة المالك أو إعدادات الصلاة.
- احسب الصلاة القادمة والوقت المتبقي بناءً على البيانات المتاحة.
- لا تدّع أنها أوقات رسمية دقيقة.

التاريخ الهجري:
استخدم Intl.DateTimeFormat مع islamic-umalqura إن كان مدعوماً.
إذا لم يكن مدعوماً، استخدم Seed Data قابلة للتعديل مع توضيح أنها تجريبية.

- تنبيه قبل الصلاة.
- إظهار الصلاة في الرئيسية.
- إظهار الصلاة في ستوري اليوم.

أي عملية حذف يجب أن تحتوي Confirm Dialog.

==================================================
8. Theme Engine
==================================================

ابنِ Theme Engine حقيقي، وليس مجرد تغيير لون بسيط.

الثيم الرسمي الافتراضي:
التراث التقني الفاخر

الثيمات الاختيارية:
1. السعودي النظيف.
2. الليلي الذهبي.
3. التراث النجدي.
4. الأبيض الرسمي.
5. النباتي الناعم.
6. المعماري الهادئ.
7. الصحراوي المعاصر.

كل ثيم يجب أن يحتوي على:
- اسم الثيم.
- وصف.
- صورة معاينة أو بطاقة معاينة.
- ألوان رئيسية.
- ألوان ثانوية.
- لون الخلفية.
- لون البطاقات.
- لون النص.
- نمط الأيقونات.
- نمط البطاقات.
- حالة: مجاني / مميز / VIP.
- زر معاينة.
- زر تطبيق.
- علامة الثيم الحالي.

المستخدم يستطيع:
- معاينة الثيم.
- تطبيق الثيم.
- الرجوع للثيم الافتراضي.
- اختيار الوضع الفاتح أو الداكن إن أمكن.
- اختيار كثافة العرض: مريح، متوسط، مضغوط.

شرط إلزامي:
تطبيق الثيم يجب أن يغير الخلفية، البطاقات، النصوص، الأزرار، الأيقونات/النمط، وليس لوناً واحداً فقط.
يجب حفظ الثيم المختار بعد إعادة التحميل قدر الإمكان.

==================================================
9. لوحة المالك المصغرة
==================================================

ابنِ لوحة مالك داخل المسار:
/admin

يجب أن تحتوي على وضع مالك تجريبي واضح:
Demo Admin Mode

لا تجعل /admin ظاهراً كجزء طبيعي من تبويبات المستخدم.

أقسام لوحة المالك:
1. Dashboard.
2. إدارة الأعضاء.
3. إدارة المواعيد العامة.
4. إدارة الرواتب والدعم.
5. إدارة رسائل اليوم.
6. إدارة ستوري اليوم.
7. إدارة الثيمات.
8. إدارة الإشعارات.
9. إدارة الأخبار والوظائف.
10. التقارير.
11. الصلاحيات.

كل قسم يجب أن يحتوي قدر الإمكان على:
- عرض.
- إضافة.
- تعديل.
- حذف.
- بحث.
- تفعيل / تعطيل.
- حالة العنصر.
- حفظ التغييرات.

شرط إلزامي:
تعديلات لوحة المالك يجب أن تظهر في واجهة المستخدم في:
- المواعيد العامة.
- الرواتب والدعم.
- رسالة اليوم.
- قوالب ستوري اليوم.
- الثيمات.
- الأخبار والوظائف.
- الإشعارات الداخلية.

لوحة المالك ليست ديكوراً.
ممنوع أن تكون مجرد صفحة شكلية.

==================================================
10. Data Layer
==================================================

أنشئ Data Layer داخلي منظم وقابل للتحويل لاحقاً إلى Supabase.

الجداول/المجموعات المطلوبة منطقياً:
- users
- user_profiles
- admin_users
- financial_events
- public_events
- user_appointments
- appointment_categories
- prayer_settings
- daily_messages
- story_templates
- themes
- notifications
- notification_preferences
- news
- jobs
- complaints
- audit_logs
- settings

أضف Seed Data عربية واقعية:
- رسائل يومية.
- مواعيد مالية.
- دعم.
- أخبار.
- وظائف.
- ثيمات.
- قوالب ستوري.
- مواعيد تقويم.
- إعدادات صلاة.
- إشعارات.

استخدم LocalStorage أو Store داخلي مناسب لحفظ التغييرات أثناء التجربة.

ممنوع أن تكون البيانات المهمة ثابتة داخل مكونات الواجهة فقط.

صمّم البنية بحيث يمكن لاحقاً ربطها بـ Supabase مع Row Level Security.

==================================================
11. Audit Logs
==================================================

أضف audit_logs أولية للعمليات المهمة من لوحة المالك.

العمليات التي يجب تسجيلها:
- إضافة موعد عام.
- تعديل موعد عام.
- حذف موعد عام.
- تعديل رسالة اليوم.
- إرسال إشعار.
- تعديل ثيم.
- تفعيل أو تعطيل عنصر.
- تعديل خبر أو وظيفة.
- تعديل رواتب أو دعم.

كل سجل يحتوي على:
- نوع العملية.
- اسم العنصر.
- الوقت.
- المستخدم أو المالك التجريبي.
- وصف مختصر.

==================================================
12. Notification Center
==================================================

ابنِ مركز إشعارات داخلي.

أنواع الإشعارات:
- إشعار قبل الراتب.
- إشعار قبل الدعم.
- إشعار قبل الفاتورة.
- إشعار قبل الموعد الشخصي.
- إشعار الصلاة القادمة.
- إشعار ستوري اليوم.
- إشعار خبر مهم.
- إشعار وظيفة جديدة.
- رسالة من المالك.

المزايا:

- عرض سجل الإشعارات.
- إرسال إشعار تجريبي من لوحة المالك.
- معاينة نص الإشعار.
- حالة مقروء / غير مقروء.
- عداد إشعارات غير مقروءة.
- بنية جاهزة للربط لاحقاً مع Firebase Cloud Messaging.

شرط إلزامي:
إرسال إشعار تجريبي من لوحة المالك يجب أن يظهر فوراً في مركز إشعارات المستخدم.

==================================================
13. سلاسل الأثر الإلزامية
==================================================

لا يكفي تنفيذ الوظيفة داخل صفحة واحدة.
يجب أن يظهر أثر البيانات في الأماكن المرتبطة.

أ. الموعد الشخصي:
- يضاف من التقويم أو الإضافة السريعة.
- يظهر في التقويم.
- يظهر في قائمة المواعيد القادمة في الرئيسية.
- يمكن تعديله.
- يمكن حذفه.
- يبقى بعد إعادة التحميل قدر الإمكان.

ب. رسالة اليوم:
- تُدار من لوحة المالك.
- تظهر في الرئيسية.
- تظهر في ستوري اليوم.
- لا تتكرر قدر الإمكان من Seed Data.

ج. الرواتب والدعم:
- تُدار من لوحة المالك.
- تظهر في مركز المال.
- تظهر في بطاقة "كم تبقى على" في الرئيسية.
- تظهر في ستوري اليوم.
- تعمل معها العدادات.

د. الثيمات:
- تُدار من Theme Engine.
- يمكن معاينتها.
- يمكن تطبيقها.
- تغير الخلفية والبطاقات والنصوص والأزرار والنمط العام.
- لا تقتصر على لون واحد.
- تبقى بعد إعادة التحميل قدر الإمكان.

هـ. الإشعارات:
- يمكن إرسال إشعار تجريبي من لوحة المالك.
- يظهر في مركز الإشعارات للمستخدم.
- يظهر كغير مقروء.
- يوجد عداد إشعارات غير مقروءة.
- يمكن تغيير حالته إلى مقروء.

و. الأخبار والوظائف:
- تُدار من لوحة المالك.
- تظهر في مركز الأخبار والوظائف.
- يمكن البحث أو الفلترة إن وُجدت بيانات متعددة.

ز. ستوري اليوم:
- يأخذ التاريخ ورسالة اليوم والعدادات من Data Layer.
- يسمح بتعديل النص.
- يسمح باختيار العناصر الظاهرة.
- يسمح بالمعاينة.
- يسمح بالحفظ الداخلي أو النسخ/المشاركة النصية حسب قدرة البيئة.

==================================================
16. الاختبار الإجباري قبل التسليم
==================================================

قبل إعلان الاكتمال، نفّذ اختباراً عملياً كالتالي:

أ. اختبار التنقل:
- افتح كل تبويب.
- افتح كل مركز.
- افتح /admin.
- تأكد أنه لا توجد صفحة فارغة.

ب. اختبار الرئيسية:
- التاريخ الهجري والميلادي يظهران.
- رسالة اليوم تظهر.
- مواقيت الصلاة تظهر.
- العدادات تعمل.
- التبويبات الخمسة تعمل.

ج. اختبار الموعد:
- أضف موعداً.
- عدله.
- احذفه.
- تأكد من ظهوره في التقويم والرئيسية.

د. اختبار التقويم:
- عرض يومي.
- عرض أسبوعي.
- عرض شهري.
- فلترة حسب التصنيف.

هـ. اختبار المال:
- افتح الرواتب.
- افتح الدعم.
- افتح الفواتير.
- أضف بنداً مخصصاً.
- تأكد من العدادات.
- جرّب الحاسبات.
- تأكد أن الحاسبات تنتج نتائج.

و. اختبار ستوري اليوم:
- افتح المعاينة.
- غيّر النص.
- غيّر العناصر الظاهرة.
- انسخ أو شارك أو احفظ داخلياً.

ز. اختبار الثيم:
- عاين ثيماً.
- طبقه.
- أعد تحميل الصفحة.
- تأكد أن الثيم ما زال محفوظاً.

ح. اختبار الإشعارات:
- أرسل إشعاراً من لوحة المالك.
- افتح مركز الإشعارات.
- تأكد أنه غير مقروء.
- اجعله مقروءاً.
- تأكد أن العداد تغيّر.

ط. اختبار لوحة المالك:
- أضف موعداً عاماً.
- عدل موعداً عاماً.
- احذف موعداً عاماً.
- عدل رسالة اليوم.
- تأكد أنها ظهرت في الرئيسية وستوري اليوم.
- غيّر الثيم.
- أرسل إشعاراً داخلياً.
- عدل خبراً أو وظيفة.
- تأكد أنه ظهر في المركز المناسب.
- راجع audit_logs.

ي. اختبار الأمان والبنية:
- /admin منفصل.
- وضع مالك تجريبي واضح.
- تسجيل خروج.
- حذف حساب مع تأكيد.
- لا صفحات مكسورة.
- لا أزرار بلا وظيفة.
- الواجهة عربية RTL بالكامل.

ك. اختبار البناء:
شغّل أوامر التحقق المتاحة:
- npm run build
- npm run lint إن وجد
- npm run test إن وجد

أي خطأ يجب إصلاحه قبل التسليم.

==================================================
17. شروط منع التسليم
==================================================

ممنوع إعلان المشروع مكتمل إذا تحقق أي بند من التالي:

- يوجد زر لا يعمل.
- توجد صفحة فارغة.
- توجد بطاقة لا تفتح شيئاً وهي يفترض أن تفتح.
- التقويم لا يدعم إضافة/تعديل/حذف.
- المواعيد لا تظهر في الرئيسية.
- المال لا يحتوي عدادات فعلية.
- الحاسبات لا تنتج نتائج.
- ستوري اليوم لا يعرض معاينة.
- الثيم لا يغير كامل النمط البصري.
- الإشعارات لا تظهر في مركز الإشعارات.
- لوحة المالك لا تغير بيانات تظهر للمستخدم.
- البيانات تضيع بعد إعادة التحميل رغم إمكانية حفظها محلياً.
- الواجهة ليست RTL بالكامل.
- يوجد TODO أو placeholder ظاهر للمستخدم.
- يوجد خطأ build.
- يوجد ادعاء تكامل خارجي غير منفذ.
- يوجد ادعاء بيانات رسمية وهي Seed Data.
