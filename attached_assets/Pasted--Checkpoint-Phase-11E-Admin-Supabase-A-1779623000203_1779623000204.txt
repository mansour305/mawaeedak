1. الهدف التنفيذي
==================================================

حوّل حماية /admin من demo guard إلى Supabase Auth guard فعلي.

المطلوب:
1. افحص src/lib/auth.ts.
2. افحص AdminLayout.
3. افحص مسار /admin وصفحة تسجيل دخول المالك.
4. اربط تسجيل دخول المالك بـ Supabase Auth عند توفر المفاتيح.
5. حافظ على demo fallback فقط عند غياب مفاتيح Supabase.
6. تحقق من session عبر Supabase.
7. تحقق من دور المستخدم من:
   - user_roles
   - roles
   - admin_users إن كان مستخدماً في البنية
8. اسمح بالدخول إلى /admin فقط إذا كان الدور:
   - admin
   - super_admin
   - content_manager
   - finance_manager
   حسب صلاحيات القسم.
9. امنع user العادي أو unauthenticated من دخول /admin.
10. لا تستخدم service_role.
11. لا تضع كلمات مرور أو أسرار في الكود.
12. لا تغيّر UI إلا بقدر بسيط جداً لتوضيح تسجيل الدخول الحقيقي.
13. حدّث QA_REPORT بالحالة الحقيقية.

6. المتطلبات التفصيلية
==================================================

أ. Auth service:
- signInWithPassword أو المسار المناسب الموجود في auth.ts.
- signOut.
- getSession.
- onAuthStateChange إن كان مناسباً.
- قراءة المستخدم الحالي.
- fallback demo عند غياب env فقط.

ب. Role check:
- بعد login اقرأ role من Supabase.
- اربط المستخدم بـ roles/user_roles.
- إن كان الدور super_admin يسمح بكل /admin.
- إن كان admin يسمح بالأقسام الإدارية الأساسية.
- إن كان content_manager يسمح بالمحتوى فقط إن كان التقسيم موجوداً.
- إن كان finance_manager يسمح بالمال فقط إن كان التقسيم موجوداً.
- إن لم تكن الصلاحيات التفصيلية جاهزة، طبق الحد الأدنى:
  super_admin/admin فقط يدخلان /admin، ووثق الباقي كقيد.

ج. AdminLayout:
- لا يعتمد على localStorage عندما Supabase env موجود.
- يستخدم session + role check.
- يعرض Login عند عدم تسجيل الدخول.
- يمنع الدخول عند عدم الصلاحية.
- يسمح بالدخول بعد تحقق role.

د. Login UX:
- لا تضع كلمة مرور ثابتة.
- المستخدم يدخل بريده وكلمة مروره.
- يمكن استخدام hrq@hotmail.com كإيميل مقترح فقط بدون كلمة مرور.
- لا تحفظ كلمة المرور في localStorage.
- لا تطبعها في console.
