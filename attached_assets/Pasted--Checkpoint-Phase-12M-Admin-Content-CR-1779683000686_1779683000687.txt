0. السياق الحالي
==================================================

تم اعتماد Phase 12L:
- Admin News CRUD أصبح Gateway.
- Admin Jobs CRUD أصبح Gateway.
- CentersNewsPage وCentersJobsPage يعكسان تغييرات Supabase.
- mode=api يعمل.

- API وPostgreSQL ما زالا موجودين.
- build ناجح.
- typecheck ناجح.

الحالة الحالية للبيانات:
- Supabase يحتوي 40 صفاً.
- themes = 10
- story_templates = 2
- daily_messages = 8

الصفحات التي تقرأ حالياً من Gateway جزئياً:
- AccountPage theme picker يقرأ themes عبر Gateway.
- StoryPage templates يقرأ story_templates عبر Gateway.
- بعض daily_messages لا تزال عبر Orval/API حسب endpoint.

القيد الحالي:
- Admin CRUD للثيمات وقوالب الستوري ورسائل اليوم ما زال غالباً عبر Orval/API.
- هذا قد يسبب divergence عند mode=supabase لأن بعض صفحات المستخدم تقرأ من Supabase بينما admin يكتب إلى API.

4. تدقيق إلزامي قبل التنفيذ
==================================================

قبل أي تعديل جديد، نفّذ تدقيقاً سريعاً على Phase 12L:

1. تحقق أن news/jobs في mode=supabase يستخدمان المعرّف الصحيح عند update/delete.
2. إذا كانت الواجهة تستخدم legacy integer id، يجب أن يستخدم Supabase write:
   - legacy_id
   أو mapping موثّق وآمن.
3. إذا كان Supabase write يستخدم uuid id فعلاً، وثّق ذلك.
4. لا تعدّل news/jobs إلا إذا وجدت خطأ فعلياً.
5. وثّق النتيجة في QA_REPORT.

1. AdminThemes الحالي.
2. AdminStory أو AdminStoryTemplates الحالي.
3. AdminMessages / DailyMessages admin الحالي.
4. hooks الحالية المستخدمة للقراءة والكتابة.
5. dataGateway.ts.
6. supabaseData.ts.
7. شكل بيانات themes في API مقابل Supabase.
8. شكل بيانات story_templates في API مقابل Supabase.
9. شكل بيانات daily_messages في API مقابل Supabase.
10. هل id في الواجهة يعتمد على legacy_id أو uuid.
11. ما الحقول المطلوبة في add/edit/toggle/delete.
12. هل RLS يسمح لـ admin/super_admin بإدارة هذه الجداول.

14. كيف سنمنع divergence في mode=supabase.

- updateThemeInSupabase

- setDefaultThemeInSupabase إن كان موجوداً في النظام

Story Templates:
- createStoryTemplateInSupabase
- updateStoryTemplateInSupabase
- deleteStoryTemplateInSupabase
- toggleStoryTemplateInSupabase

Daily Messages:
- createDailyMessageInSupabase
- updateDailyMessageInSupabase
- deleteDailyMessageInSupabase
- toggleDailyMessageInSupabase أو equivalent حسب schema

ب. Data Gateway:
أضف في dataGateway.ts functions مقابلة:
- gwUpdateTheme / gwToggleTheme / gwSetDefaultTheme
- gwCreateStoryTemplate / gwUpdateStoryTemplate / gwDeleteStoryTemplate / gwToggleStoryTemplate
- gwCreateDailyMessage / gwUpdateDailyMessage / gwDeleteDailyMessage / gwToggleDailyMessage

ج. السلوك حسب mode:
- api:
  API فقط.
- supabase_shadow:
  API فقط.
- supabase:
  Supabase فقط لنطاق Phase 12M.
  لا fallback صامت في write.

د. Query invalidation:
بعد أي عملية:
- حدّث gwQueryKeys.themes
- حدّث gwQueryKeys.storyTemplates
- حدّث gwQueryKeys.dailyMessages إن وجد
- حدّث Orval query keys للتوافق إذا بقيت بعض الصفحات API.
- لا تترك stale cache.

هـ. IDs:
- إذا واجهة admin تستخدم legacy integer id، استخدم legacy_id في Supabase.
- لا تكسر واجهة المستخدم بسبب uuid.
- وثّق mapping المستخدم.
