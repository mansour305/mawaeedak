==================================================
0. السياق الحالي
==================================================

تم اعتماد Phase 12J:
- NotificationsPage أصبحت تستخدم Gateway للقراءة.
- mark-read يستخدم Gateway.
- mark-all-read يستخدم Gateway.
- delete notification يستخدم Gateway.
- mode=api يعمل عبر API.
- mode=supabase_shadow يعمل عبر API ولا يكتب إلى Supabase.
- mode=supabase يقرأ ويكتب للإشعارات عبر Supabase.
- لا fallback صامت في write عند mode=supabase.
- build ناجح.
- typecheck ناجح.
- لا service_role.
- لا أسرار hardcoded.
- لا DROP/TRUNCATE/DELETE.
- Supabase Auth و/admin سليمان.

القيود المتبقية في نظام الإشعارات:
1. TopBar unread count ما زال يقرأ من API/Orval.
2. AdminNotifications send/delete ما زالت على API/Orval.
3. عند mode=supabase قد يحدث اختلاف بين:
   - NotificationsPage من Supabase.
   - TopBar count من API.
   - AdminNotifications من API.

المطلوب الآن إغلاق هذا التباين داخل نطاق الإشعارات فقط.

7. عند mode=supabase:
   - NotificationsPage + TopBar count + AdminNotifications notifications operations تستخدم Supabase بقدر النطاق الآمن.
8. لا تحوّل أي نطاق خارج الإشعارات.
9. لا تحوّل Calendar.
10. لا تحوّل Finance.

==================================================
5. متطلبات التنفيذ
==================================================

نفّذ بأقل تغيير آمن:

أ. TopBar unread count:
- أضف Gateway function مثل:
  gwGetUnreadNotificationsCount
- عند mode=api أو shadow:
  استخدم API الحالي.
- عند mode=supabase:
  احسب العدد من Supabase:
  user_id = current user
  is_read = false
- أضف fallback قراءة إلى API إذا فشل Supabase read فقط.
- لا تكسر TopBar.

ب. AdminNotifications read:
- اجعل قراءة قائمة الإشعارات تستخدم Gateway إن كان آمناً.
- عند mode=api/shadow:
  API.
- عند mode=supabase:
  Supabase.

ج. AdminNotifications send:
- إذا كان schema واضحاً وRLS يسمح:
  أضف Gateway sendNotification.
- في mode=supabase:
  أنشئ notification في Supabase للمستخدم الحالي hrq أو current session user.
- لا تستخدم service_role.
- لا تضع recipient عام غير واضح.
- إذا لم يكن آمنًا:
  اتركه API ووثّق السبب، ولا تدّع اكتماله.

د. AdminNotifications delete:
- استخدم gwDeleteNotification إذا مناسب.
- عند mode=supabase:
  احذف من Supabase وفق legacy_id/user_id.
- لا fallback صامت في write.

هـ. Cache/refresh:
- بعد أي write:
  حدّث NotificationsPage.
  حدّث TopBar count.
  حدّث AdminNotifications.
- تجنب stale count.
