==================================================
0. السياق الحالي
==================================================

تم اعتماد Phase 12N:
- CalendarPage appointments أصبح Gateway.
- HomePage upcoming appointments أصبح Gateway.
- create/update/delete appointments أصبح Gateway.
- mode=api يعمل.
- mode=supabase_shadow يعمل.
- mode=supabase يعمل للمواعيد.
- لا fallback صامت في write عند mode=supabase.
- FinancePage و financial_events لم تُلمس.
- API وPostgreSQL ما زالا موجودين.
- default data source = api.
- build ناجح.
- typecheck ناجح.
- لا service_role.
- لا أسرار hardcoded.
- لا DROP/TRUNCATE/DELETE غير مقصود.

حالة بيانات Supabase:
- financial_events = 8
- financial_events مرتبطة بـ user_id الخاص بـ hrq@hotmail.com.
- legacy_id محفوظ.
- migration_batch محفوظ.
- amount قد يكون numeric في Supabase ويحتاج Number() في الواجهة.

القيد الحالي:
- FinancePage CRUD ما زال Orval/API.
- HomePage financial countdown غالباً ما زال API.
- StoryPage counters غالباً ما زالت API.
- Admin financial events إن وجدت غالباً ما زالت API.
- عند mode=supabase، أي صفحة تقرأ financial_events من Supabase بينما تكتب API قد تسبب divergence.

==================================================
1. الهدف التنفيذي
==================================================

حوّل financial_events فقط إلى Gateway:

1. FinancePage list/read financial_events.
2. FinancePage add financial event.
3. FinancePage edit financial event.
4. FinancePage delete financial event.
5. FinancePage countdown أو أي قراءة مالية محسوبة.
6. HomePage financial countdown.
7. StoryPage financial counters.
8. Admin financial events CRUD إن كان موجوداً ومباشراً.

المطلوب حسب mode:

أ. mode=api:
- القراءة والكتابة عبر API كما كانت.
- لا تغيير سلوك المستخدم.

ب. mode=supabase_shadow:
- القراءة والكتابة عبر API.
- لا كتابة إلى Supabase.
- يمكن إجراء shadow comparison فقط.

ج. mode=supabase:
- قراءة financial_events من Supabase.
- إضافة financial_event إلى Supabase.
- تعديل financial_event في Supabase.
- حذف financial_event من Supabase.
- HomePage countdown يقرأ من نفس مصدر FinancePage قدر الإمكان.
- StoryPage counters تقرأ من نفس مصدر FinancePage قدر الإمكان.
- لا fallback صامت إلى API في write.
- عند فشل write، أظهر toast/error واضح.

ممنوع تحويل:
- appointments لأنها اكتملت في Phase 12N ولا تُلمس إلا إذا لزم invalidation.
- notifications/news/jobs/themes/story_templates/daily_messages لأنها منتهية.
- profiles.
- audit_logs.
- أي نطاق جديد خارج financial_events.

4. تدقيق إلزامي قبل التنفيذ
==================================================

   - financial_events countdown
   - create financial_event
   - update financial_event
   - delete financial_event

3. افحص HomePage وأي مكون يستخدم financial countdown.
4. افحص StoryPage وأي مكون يستخدم العدادات المالية.
5. افحص Admin financial events إن وجد.
6. افحص dataGateway.ts.
7. افحص supabaseData.ts.
8. افحص شكل financial_event في API مقابل Supabase.
9. افحص هل UI يستخدم id كـ legacy integer أو uuid.
10. افحص fields:
   - name/title
   - amount
   - date
   - type/category
   - emoji إن وجد
   - is_active/status إن وجد
   - user_id
   - legacy_id
   - created_at/updated_at

11. افحص endpoint:
   - /api/financial-events/countdown
   وهل يحسب days/remaining server-side.

12. إذا كان countdown server-calculated:
   - عند mode=supabase يجب إما إعادة نفس الحساب بدقة في Gateway.
   - أو إبقاء countdown على API مع توثيق صريح أن التحويل جزئي.
   - لا تقبل cutover كامل إذا الحساب غير مطابق.

13. حدد mapping واضح:
   - id في الواجهة = legacy_id إن كان legacy موجوداً.
   - عند إنشاء صف جديد في Supabase، ما قيمة legacy_id؟
   - إذا legacy_id مطلوب لتوافق الواجهة، ضع استراتيجية آمنة:
     أ. إما توليد legacy_id جديد من max(legacy_id)+1 داخل Supabase.
     ب. أو استخدام Supabase id وإرجاع id متوافق موثق.
   - لا تنفذ شيئاً حتى توثق القرار.

14. افحص RLS:
   - المستخدم يرى أحداثه المالية فقط.
   - المستخدم يضيف أحداثه المالية فقط.
   - المستخدم يعدّل أحداثه المالية فقط.
   - المستخدم يحذف أحداثه المالية فقط.

15. حدد استراتيجية invalidation بعد كل mutation:
   - Finance list.
   - Finance countdown.
   - HomePage financial counters.
   - StoryPage counters.
   - أي Orval keys متبقية للتوافق.
