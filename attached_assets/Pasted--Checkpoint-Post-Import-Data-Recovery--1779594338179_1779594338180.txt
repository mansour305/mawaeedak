استمر من آخر Checkpoint. لا تعيد البناء من البداية.

المرحلة الحالية:
Post-Import Data Recovery + Seed Integrity Repair + Data Source Verification

الدور:
تصرف كمهندس Backend/Data + QA + Product Reliability داخل Replit Agent.
مهمتك ليست إضافة ميزات، وليست تحسين التصميم، وليست إعادة بناء التطبيق. مهمتك الوحيدة الآن هي إصلاح مشكلة البيانات الفارغة بعد نقل مشروع "مواعيدك" إلى حساب hrq، مع الحفاظ الكامل على المشروع كما هو.

==================================================
0. السياق الحالي
==================================================

تم نقل مشروع "مواعيدك" من حساب سابق إلى حساب hrq عن طريق ZIP.

بعد النقل:
- تم حل مشكلة node_modules missing.
- تم حل مشكلة vite command not found.
- التطبيق يعمل في Preview.
- تم إنشاء أو تطبيق schema / database tables.
- لكن الواجهة تعرض بيانات فارغة.

المشكلة الظاهرة في الصفحة الرئيسية:
- رسالة اليوم لا تظهر أو تظهر: "لا توجد رسالة اليوم".
- مواقيت الصلاة لا تظهر أو تظهر: "لا توجد بيانات".
- بطاقة "كم باقي على" لا تعرض العدادات أو تظهر: "لا توجد عدادات حالياً".

الاحتمالات الرئيسية:
- ZIP لم ينقل بيانات runtime أو local database.
- الجداول موجودة لكنها فارغة.
- Seed Data لم تُنفذ بعد النقل.
- Database Initialization ناقص.
- API يرجع 200 لكن ببيانات فارغة.
- API يرجع 500 لبعض الجداول.
- الواجهة تقرأ من مصدر مختلف عن المصدر المزروع.
- لوحة المالك تكتب في مصدر والواجهة تقرأ من مصدر آخر.
- fallback/local service لا يحتوي بيانات أساسية بعد النقل.
- أسماء الجداول أو الحقول لا تطابق ما تتوقعه الواجهة.
- فلاتر status / is_active / source_type تمنع ظهور البيانات.
- تواريخ الأحداث المالية قديمة لذلك العدادات لا تظهر.

==================================================
1. الهدف التنفيذي
==================================================

أعد التطبيق إلى حالة Publishable Preview صحيحة بعد النقل.

يجب أن تظهر البيانات الأساسية فعلياً في الواجهة من مصدر البيانات الصحيح داخل المشروع، وليس عبر نصوص ثابتة داخل المكونات.

المطلوب تحديداً:
1. تشخيص سبب فراغ البيانات.
2. إصلاح Seed Data أو Database Initialization أو Service Layer حسب السبب الحقيقي.
3. إعادة ظهور البيانات الأساسية في:
   - الرئيسية.
   - مركز المال.
   - ستوري اليوم.
   - مركز الإشعارات إن كان منفذاً.
   - لوحة المالك /admin.
4. التأكد أن الواجهة ولوحة المالك تقرأ وتكتب من نفس مصدر البيانات.
5. تشغيل build بنجاح.
6. الحفاظ على Preview يعمل.
7. توثيق ما حدث في QA_REPORT أو التقرير المناسب.

المخرج المقبول:
- Publishable Preview إذا ظهرت البيانات الأساسية وعمل build.
- Needs Fixes إذا بقي خلل جوهري مع سبب واضح لا يمكن تجاوزه داخل البيئة الحالية.

==================================================
3. ممنوعات صارمة
==================================================

ممنوع نهائياً:
- لا تعيد بناء المشروع من الصفر.
- لا تغيّر Routing.
- لا تغيّر التصميم أو Visual Theme.
- لا تضف ميزات جديدة.
- لا تضف Supabase الآن.
- لا تضف Firebase الآن.
- لا تضغط Publish.
- لا تضع البيانات مباشرة داخل React components.
- لا تستخدم hardcoded data داخل pages/components كحل سريع.
- لا تخفي المشكلة بعرض نص ثابت في الواجهة.
- لا تحذف الجداول أو ملفات البيانات إلا بسبب واضح وموثق.
- لا تغيّر API إلا إذا ثبت أن الخلل في الربط.
- لا تغيّر Data Layer إلا بالقدر الضروري لإصلاح seed/initialization/reading.
- لا تجعل seed يمسح بيانات موجودة.
- لا تجعل seed يكرر البيانات عند إعادة التشغيل.
- لا تستخدم تواريخ قديمة تجعل العدادات فارغة.
- لا تعرض عدادات سالبة.
- لا تدّع أن البيانات رسمية إذا كانت seed أو fallback أو admin_managed.
- لا تكسر Preview.
- لا تكسر build.
- لا تكتب في QA_REPORT شيئاً لم تختبره فعلاً.
- لا تغيّر package.json أو lockfile إلا إذا كان ضرورياً لإصلاح تشغيل فعلي.
- لا تضف مكتبة جديدة إلا إذا كان الإصلاح مستحيلاً بدونها واذكر السبب.

المسموح:
- إصلاح seed scripts.
- إصلاح database initialization.
- إصلاح fallback/local data service.
- إصلاح service layer.
- إضافة Seed Data مفقودة.
- استخدام upsert آمن.
- إضافة seed script واضح إذا كان مفقوداً.
- إصلاح API endpoints التي ترجع بيانات فارغة أو 500.
- تحديث QA_REPORT بما حدث فعلاً.

==================================================
4. بروتوكول العمل قبل أي تعديل
==================================================

قبل أي تعديل، نفّذ Baseline سريع:

1. اقرأ package.json وحدد أوامر التشغيل الفعلية.
2. حدد مدير الحزم المستخدم: pnpm أو npm أو yarn.
3. افحص بنية data/services/API.
4. افحص أين تُقرأ بيانات الرئيسية.
5. افحص أين تُقرأ بيانات المال.
6. افحص أين تُقرأ بيانات ستوري اليوم.
7. افحص أين تكتب لوحة المالك.
8. افحص هل هناك قاعدة بيانات محلية أو PostgreSQL أو Drizzle أو Prisma أو fallback service.
9. افحص هل هناك seed script موجود.
10. افحص هل هناك migration/schema فقط بدون seed.
11. سجّل السبب المرجح قبل التعديل.

لا تبدأ بأي إصلاح قبل تحديد سبب منطقي واضح.

==================================================
5. بروتوكول التشخيص الجذري Root Cause
==================================================

افحص بالترتيب:

1. هل الجداول أو التخزين الحالي موجود؟
2. هل الجداول موجودة لكنها فارغة؟
3. هل API يرجع 200 مع arrays فارغة؟
4. هل API يرجع 500؟
5. هل schema أُنشئ بدون insert / seed data؟
6. هل توجد ملفات seed لكنها لم تُنفذ بعد النقل؟
7. هل توجد fallback data لكنها غير مربوطة بالواجهة؟
8. هل الواجهة تقرأ من جدول مختلف عن الجدول المزروع؟
9. هل لوحة المالك تكتب إلى مصدر والواجهة تقرأ من مصدر آخر؟
10. هل أسماء الجداول أو الحقول لا تطابق schema؟
11. هل المشكلة بسبب أن ZIP لم ينقل بيانات runtime أو local database؟
12. هل هناك migration أو script لم يُنفذ بعد الاستيراد؟
13. هل هناك service interface موحد أم أن كل صفحة تقرأ من مصدر مختلف؟
14. هل هناك اختلاف بين مصدر بيانات الرئيسية والمال وستوري اليوم؟
15. هل توجد فلاتر status/is_active/source_type تمنع ظهور البيانات رغم وجودها؟
16. هل التواريخ قديمة مما يجعل العدادات تظهر فارغة؟
17. هل المشكلة بسبب timezone أو UTC؟
18. هل seed موجود لكن لا يعمل تلقائياً عند تشغيل dev server؟
19. هل API يعتمد على database بينما fallback يحتوي البيانات أو العكس؟

لا تعالج الأعراض فقط. أصلح السبب الذي يجعل البيانات لا تظهر.

==================================================
6. قاعدة الإصلاح الآمن
==================================================

نفّذ أقل تعديل آمن يحقق الإصلاح.

إذا كانت الجداول فارغة:
- ازرع Seed Data أساسية.
- استخدم upsert إن كان متاحاً.
- استخدم keys أو slugs أو codes ثابتة لمنع التكرار.
- لا تستخدم insert عشوائي يضاعف البيانات مع كل تشغيل.
- لا تمسح بيانات موجودة.
- لا تجعل seed destructive.
- اجعل seed قابلاً لإعادة التشغيل بدون مضاعفة السجلات.

إذا كان الربط مكسوراً:
- أصلح Service Layer أو API بحيث تقرأ الواجهة من المصدر الصحيح.
- لا تنقل البيانات إلى component.
- لا تبنِ مساراً جديداً موازياً بلا حاجة.
- اجعل الواجهة ولوحة المالك تستخدمان نفس مصدر البيانات.

إذا كان fallback هو المستخدم:
- تأكد أن fallback يحتوي بيانات أساسية.
- تأكد أن fallback موثق كحل مؤقت.
- لا تدّع أنه Backend إنتاجي.

إذا كانت المشكلة في التواريخ:
- أصلح منطق المواعيد القادمة.
- استخدم تواريخ مستقبلية أو recurrence.
- لا تعرض عدادات سالبة.
- لا تجعل "كم باقي على" فارغة بسبب تاريخ قديم.

إذا احتجت تعديل schema:
- عدّل بأضيق نطاق ممكن.
- لا تكسر الجداول الحالية.
- وثّق سبب التعديل.
- تأكد أن API والواجهة ما زالت تعمل.

==================================================
7. مصدر الحقيقة الواحد
==================================================

يجب أن يكون لكل نوع بيانات مصدر واضح.

القاعدة:
- الواجهة تقرأ من Service Layer.
- لوحة المالك تكتب إلى نفس Service Layer.
- seed data يغذي نفس المصدر الذي تقرأ منه الواجهة.
- المكونات لا تحمل بيانات أساسية داخلها.
- API لا يرجع arrays فارغة إذا كانت seed موجودة.
- إذا عُدلت رسالة اليوم من /admin، تظهر في الرئيسية وستوري اليوم.
- إذا عُدل حدث مالي من /admin، يظهر في المال والرئيسية وستوري اليوم.
- إذا أُرسل إشعار من /admin، يظهر في مركز الإشعارات.

إذا وجدت أكثر من مصدر بيانات لنفس الشيء:
- لا تضف مصدراً ثالثاً.
- وحّد القراءة والكتابة على أقرب Service Layer موجود.
- وثّق مصدر الحقيقة في التقرير.

==================================================
8. البيانات الأساسية المطلوبة
==================================================

أعد توفير الحد الأدنى من البيانات التالية حتى لا تظهر الواجهة فارغة.

أولاً: daily_messages
- أضف 8 رسائل يومية عربية على الأقل.
- status = active أو ما يعادله.
- رسالة واحدة فعالة يجب أن تظهر في الرئيسية.
- نفس الرسالة يجب أن تظهر في ستوري اليوم.
- لا تجعل الرسائل داخل component.

رسائل مقترحة:
- من رتّب يومه ملك وقته.
- الصبر مفتاح الفرج، وما أقرب الفرج من أهل الصبر.
- يومك يبدأ بقرار، فاجعله قراراً نافعاً.
- الوقت إذا حُفظ أثمر.
- رتّب مواعيدك، ترتّب أولوياتك.
- كل موعد له وقته، وكل وقت له قيمة.
- خذ من يومك ما يعينك على غدك.
- البداية المنظمة تختصر نصف الطريق.

ثانياً: prayer_times أو prayer_settings
- المدينة الافتراضية: الرياض.
- المنطقة الزمنية: Asia/Riyadh.
- أوقات الصلاة الأساسية:
  الفجر، الشروق، الظهر، العصر، المغرب، العشاء.
- يجب أن تظهر مواقيت الصلاة في الرئيسية.
- يجب حساب الصلاة القادمة والوقت المتبقي.
- إذا كانت البيانات seed أو fallback، لا تعرضها كبيانات رسمية.
- استخدم source_type = admin_managed أو fallback إذا كان الحقل موجوداً.
- استخدم source_name = "بيانات مدارة من المنصة" إذا كان الحقل موجوداً.

ثالثاً: financial_events
أضف أحداثاً فعالة للعدادات:
- الراتب.
- حساب المواطن.
- الضمان.
- حافز.
- الدعم السكني.
- التقاعد.
- التأهيل.
- ساند / التأمينات.
- الدعم الزراعي.
- دعم ريف إن كان مدعوماً.
- فواتير أساسية إن كان لها جدول منفصل.

شروط financial_events:
- لا تستخدم تواريخ قديمة.
- استخدم تواريخ مستقبلية صالحة أو recurrence إن كان موجوداً.
- إذا كان التاريخ الشهري مرّ، احسب الموعد القادم للشهر التالي.
- لا تعرض عدادات سالبة.
- status = active أو ما يعادله.
- source_type = admin_managed أو estimated إذا كان الحقل موجوداً.
- لا تستخدم official إلا بوجود source_name و last_verified_at حقيقيين.

رابعاً: appointments أو public_events
- أضف موعداً أو موعدَين قادمين.
- يجب أن تظهر في الرئيسية إذا كانت الواجهة تعرض المواعيد القادمة.
- يجب أن تكون قابلة للإدارة من لوحة المالك إذا كان ذلك منفذاً.

خامساً: story_templates
- أضف قالب ستوري اليوم الأساسي.
- أضف قالب تراثي بسيط إن كان schema يدعم تعدد القوالب.
- يجب أن يقرأ ستوري اليوم رسالة اليوم والعدادات من Data Layer.

سادساً: notifications
- أضف إشعاراً داخلياً أو اثنين.
- يدعم read/unread إذا كان schema يدعم ذلك.
- يظهر في مركز الإشعارات.
- يظهر عداد غير المقروء إن كان منفذاً.

سابعاً: themes
- تأكد أن الثيم الافتراضي "التراث التقني الفاخر" موجود ومفعّل.
- لا تغيّر التصميم الآن.
- أصلح غياب الثيم فقط إذا كان يسبب خطأ أو يمنع تحميل الواجهة.

ثامناً: news/jobs إن وجدت
- أضف بيانات بسيطة للاختبار فقط.
- لا تدّع أنها رسمية.
- status = active إذا كان الحقل موجوداً.
- source_type = admin_managed أو fallback إذا كان الحقل موجوداً.

- الرواتب.

==================================================

==================================================
13. لوحة المالك بعد الإصلاح
==================================================

افحص /admin وتأكد من الآتي:

- بيانات رسائل اليوم تظهر.
- بيانات الرواتب والدعم تظهر.
- بيانات المواعيد العامة تظهر.
- بيانات الإشعارات تظهر أو يمكن إنشاؤها.
- بيانات الثيمات تظهر إن كانت مدعومة.
- يمكن تعديل البيانات إذا كانت الإدارة منفذة.
- يظهر أثر التعديل في واجهة المستخدم.
- العمليات الإدارية المهمة تسجل audit log إذا كان ذلك منفذاً سابقاً.

لا تبنِ لوحة مالك جديدة.
فقط أصلح البيانات والربط.

==================================================
14. الاختبار الإجباري بعد الإصلاح
==================================================

نفّذ هذه الاختبارات بعد الإصلاح:

1. افتح الرئيسية.
2. تأكد أن رسالة اليوم ظهرت.
3. تأكد أن مواقيت الصلاة ظهرت.
4. تأكد أن الصلاة القادمة ظهرت.
5. تأكد أن العدادات المالية ظهرت.
6. افتح المال وتأكد أن البيانات موجودة.
7. افتح ستوري اليوم وتأكد أنه يقرأ نفس الرسالة والعدادات.
8. افتح الإشعارات وتأكد من وجود سجل أو إمكانية إنشاء إشعار.
9. افتح /admin.
10. عدّل رسالة اليوم من /admin إن كانت الوظيفة موجودة.
11. ارجع للرئيسية وستوري اليوم وتأكد أن التعديل ظهر.
12. تأكد أنه لا توجد أخطاء 500 في API.
13. تأكد أنه لا توجد شاشة بيضاء.
14. شغّل build.
15. شغّل typecheck إن وجد.
16. تأكد أن Preview يعمل.

==================================================
15. أوامر الفحص
==================================================

شغّل حسب المتاح في package.json:

- pnpm install إذا احتاج.
- pnpm build أو npm run build حسب المشروع.
- npm run typecheck إن وجد.
- npm run lint إن وجد.
- npm run test إن وجد.

إذا كان الأمر غير موجود:
- لا تدّع أنك شغلته.
- اذكر أنه غير متوفر.

==================================================
16. معيار قبول الإصلاح
==================================================

لا تعتبر الإصلاح منتهياً إلا إذا تحقق كل ما يلي:

- الجداول أو مصدر البيانات يحتوي بيانات أساسية.
- الرئيسية لا تعرض بيانات فارغة.
- رسالة اليوم ظهرت.
- مواقيت الصلاة ظهرت.
- الصلاة القادمة ظهرت.
- العدادات المالية ظهرت.
- مركز المال يعرض بيانات.
- ستوري اليوم يقرأ من نفس البيانات.
- /admin يرى البيانات ويستطيع تعديلها إن كانت الإدارة منفذة.
- لا توجد أخطاء 500 في API.
- لا توجد شاشة بيضاء.
- لا توجد صفحات مكسورة.
- لا توجد بيانات hardcoded داخل components كحل مؤقت.
- build ناجح.
- Preview يعمل.

الحكم النهائي يجب أن يكون واحداً فقط:
Publishable Preview / Needs Fixes

==================================================
17. تحديث التوثيق
==================================================

حدّث QA_REPORT.md أو ملف التقرير المناسب بما يطابق الواقع الفعلي:

- سبب فراغ البيانات بعد النقل.
- هل كانت الجداول فارغة أم كان الربط مكسوراً؟
- ما الذي تم زرعه؟
- هل seed آمن ولا يكرر البيانات؟
- هل البيانات official أم admin_managed أم fallback أم estimated؟
- ما الاختبارات التي تم تنفيذها؟
- نتيجة build.
- نتيجة typecheck إن وجدت.
- القيود المتبقية.

لا تكتب في التوثيق شيئاً لم يتم فعلياً.
