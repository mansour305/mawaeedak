4. القرار المعماري المطلوب
==================================================

اعتمد تصميم Data Layer بهذا الأسلوب:

1. API الحالي يبقى المصدر الافتراضي.
2. Supabase Data Adapter يُبنى كمصدر موازٍ.
3. Feature flag مثل:
   VITE_DATA_SOURCE_MODE=api | supabase_shadow | supabase

4. القيمة الافتراضية يجب أن تكون:
   api

5. وضع supabase_shadow يعني:
   - التطبيق يستمر يعرض بيانات API.
   - يتم جلب Supabase للمقارنة فقط إن أمكن.
   - لا يكسر الواجهة عند فشل Supabase.

6. وضع supabase سيكون للمرحلة القادمة فقط، ولا يُفعل الآن.

7. أي خطأ في Supabase يجب ألا يكسر Preview.

   - getComplaintsFromSupabase

5. كل دالة يجب أن:
   - تتحقق من وجود Supabase client.
   - تتعامل مع الأخطاء بدون رمي كسر عام.
   - تعيد shape متوافقاً مع UI أو توثق الفروقات.
   - تستخدم session/user_id عند الحاجة.
   - لا تستخدم service_role.

   - يقارن counts بين API وSupabase.

   - لا يغير البيانات.

   - يعرض حالة مقارنة فقط.
