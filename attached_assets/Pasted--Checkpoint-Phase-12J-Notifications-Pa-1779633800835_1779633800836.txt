1. الهدف التنفيذي
==================================================

حوّل NotificationsPage فقط إلى Read+Write Consistency عبر Data Gateway.

المطلوب:
1. اجعل قراءة NotificationsPage تمر عبر Gateway عند الإمكان.
2. عند mode=api:
   - تقرأ من API.
   - mark-read / mark-all-read تكتب إلى API.
   - السلوك يبقى كما كان.

3. عند mode=supabase_shadow:
   - تقرأ من API.
   - تكتب إلى API.
   - لا تكتب إلى Supabase.
   - يمكن تشغيل shadow comparison بدون كسر.

4. عند mode=supabase:
   - تقرأ NotificationsPage من Supabase.
   - mark-read / mark-all-read تكتب إلى Supabase.
   - لا يوجد fallback صامت للكتابة إلى API إذا فشل Supabase write.
   - عند فشل الكتابة، اعرض خطأ واضح أو toast مناسب بدون كسر الصفحة.

5. لا تحوّل delete notification الآن إلا إذا كان منخفض المخاطر ومكتمل ومختبر. الأصل: اترك delete على API أو عطّله في mode=supabase مع توثيق واضح إن كان سيسبب divergence.
6. لا تحوّل send notification من admin الآن.
7. لا تحوّل Calendar أو Finance.
8. لا تحذف API.
9. لا تضغط Publish.
10. حدّث QA_REPORT والوثائق.

5. متطلبات التنفيذ
==================================================

نفّذ بأقل تغيير آمن:

1. أنشئ hook أو service مناسب مثل:
   useGatewayNotifications
   أو عدّل NotificationsPage مباشرة بأقل تغيير.

2. يجب أن يدعم:
   - loading
   - error
   - refetch/refresh بعد mark-read
   - unread count

3. عند mode=supabase:
   - list notifications من Supabase.
   - mark-read إلى Supabase.
   - mark-all-read إلى Supabase.
   - refresh من Supabase بعد النجاح.

4. عند mode=api أو supabase_shadow:
   - استعمل السلوك الحالي API/Orval قدر الإمكان.

5. delete notification:
   - لا تحوله إلى Supabase في هذه المرحلة إلا إذا كان آمن ومختبر.
   - إذا بقي API، وثّق أن delete لم يتحول.
   - إذا سبب divergence في mode=supabase، اجعل behavior واضحاً في QA_REPORT.

6. لا تكسر Header unread count.
7. لا تكسر Admin notifications.
8. لا تكسر /admin/data-layer.
