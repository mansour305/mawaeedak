استمر من آخر Checkpoint. لا تعيد البناء من البداية.

المرحلة الحالية:
Final Verification Gate بعد إصلاح بيانات النقل

الدور:
تصرف كمهندس QA + Build Verification + Product Reliability داخل Replit Agent.
مهمتك ليست إضافة ميزات، وليست تحسين التصميم، وليست إعادة بناء التطبيق. مهمتك الآن هي التحقق النهائي من أن إصلاح بيانات النقل نجح فعلياً، وأن المشروع مستقر قبل الانتقال إلى مرحلة التصميم المرجعي.

==================================================
0. السياق الحالي
==================================================

تم نقل مشروع "مواعيدك" إلى حساب hrq عن طريق ZIP.

بعد النقل حدثت مشاكل وتم التعامل معها:
- node_modules كانت مفقودة.
- vite command not found ظهر وتم إصلاحه.
- schema / database tables تم تطبيقها.
- الواجهة كانت تعرض بيانات فارغة.
- تم تنفيذ إصلاح لاستعادة البيانات الأساسية.

النتائج الظاهرة الآن حسب آخر حالة:
- رسالة اليوم ظهرت في الرئيسية.
- مواقيت الصلاة ظهرت.
- الصلاة القادمة والوقت المتبقي ظهرت.
- العدادات المالية ظهرت بتواريخ مستقبلية.
- مركز المال يعرض بيانات.
- ستوري اليوم يقرأ من Data Layer.
- /admin يرى ويعدل البيانات.
- لا توجد أخطاء 500.
- TypeScript نظيف 0 أخطاء.
- Preview يعمل.
- لا توجد بيانات hardcoded داخل components.

هذه المرحلة ليست لإضافة شيء جديد. هذه مرحلة تأكيد نهائي فقط.

==================================================
1. الهدف التنفيذي
==================================================

الهدف هو تثبيت حكم واضح ونهائي على حالة المشروع بعد إصلاح بيانات النقل:

- هل التطبيق مستقر كـ Publishable Preview؟
- هل build ناجح؟
- هل Preview يعمل؟
- هل البيانات الأساسية ما زالت ظاهرة بعد إعادة التحميل؟
- هل لا توجد أخطاء API أو صفحات مكسورة؟
- هل يمكن الانتقال بأمان إلى مرحلة Visual Polish لاحقاً؟

لا تعلن Publishable Preview إلا بعد التحقق العملي.

==================================================
2. خارج النطاق
==================================================

ممنوع في هذه المرحلة تنفيذ أي عمل خارج التحقق النهائي.

خارج النطاق:
- تحسين التصميم.
- تعديل الثيم.
- Visual Polish.
- إضافة ميزات جديدة.
- إعادة بناء المشروع.
- تغيير Routing.
- تغيير API.
- تغيير Data Layer.
- إضافة Supabase.
- إضافة Firebase.
- الضغط على Publish.
- تعديل الواجهة إلا إذا ظهر خطأ كاسر واضح.
- تغيير seed data إلا إذا اختفت البيانات أو ظهر خطأ فعلي.

==================================================
3. ممنوعات صارمة
==================================================

ممنوع نهائياً:
- لا تعيد بناء المشروع من الصفر.
- لا تضف ميزات جديدة.
- لا تغيّر التصميم.
- لا تغيّر الهوية البصرية.
- لا تغيّر API إلا إذا ظهر خطأ حقيقي يمنع التشغيل.
- لا تغيّر Data Layer إلا إذا ظهر خلل فعلي في القراءة أو الكتابة.
- لا تغيّر Routing.
- لا تضف مكتبات جديدة.
- لا تضغط Publish.
- لا تكتب في QA_REPORT شيئاً لم تختبره فعلياً.
- لا تدّع نجاح build إن لم يتم تشغيله.
- لا تدّع نجاح typecheck إن لم يتم تشغيله.
- لا تدّع أن التطبيق Production Ready.
- لا تغيّر الحكم إلى Production Ready لأن المشروع ما زال يعتمد على بيئة/fallback أو إعداد غير مكتمل إنتاجياً.
- لا تستخدم بيانات hardcoded داخل components كحل مؤقت.
- لا تخفي مشاكل البيانات بعبارات ثابتة في الواجهة.

المسموح فقط:
- تشغيل build.
- تشغيل typecheck إن وجد.
- تشغيل lint/test إن وجدت.
- فحص Preview.
- فحص API.
- فحص الصفحات الأساسية.
- فحص بقاء البيانات بعد reload.
- إصلاح خطأ كاسر صغير فقط إذا ظهر أثناء التحقق.
- تحديث QA_REPORT بما تم فعلياً.

==================================================
4. بروتوكول التحقق قبل أي تعديل
==================================================

قبل أي تعديل، نفّذ Baseline Verification:

1. اقرأ package.json وحدد أوامر الفحص المتاحة فعلياً.
2. حدّد هل المشروع يستخدم pnpm أو npm.
3. لا تشغّل أوامر غير موجودة.
4. افتح Preview.
5. افحص الصفحة الرئيسية.
6. افحص مركز المال.
7. افحص ستوري اليوم.
8. افحص /admin.
9. افحص Console أو API errors إن كانت ظاهرة.
10. سجّل أي خلل فعلي قبل محاولة إصلاحه.

لا تبدأ بإصلاحات استباقية. هذه مرحلة تحقق، وليست مرحلة تطوير.

==================================================
5. مصدر الحقيقة المطلوب تأكيده
==================================================

تأكد أن البيانات الأساسية ما زالت تأتي من Data Layer / Service Layer وليست من نصوص ثابتة داخل components.

يجب تأكيد التالي:

- رسالة اليوم تظهر من مصدر البيانات.
- ستوري اليوم يقرأ نفس رسالة اليوم.
- العدادات المالية تقرأ من نفس مصدر أحداث المال.
- مركز المال يعرض نفس بيانات العدادات/الأحداث.
- /admin يرى البيانات نفسها أو يستطيع تعديلها إن كانت الإدارة منفذة.
- لا توجد بيانات أساسية hardcoded داخل components كحل مؤقت.

إذا وجدت أي hardcoded data داخل components لإخفاء مشكلة البيانات:
- اعتبر ذلك خللاً.
- لا تحكم Publishable Preview حتى يتم إصلاحه أو توثيقه بوضوح كـ Needs Fixes.

==================================================
6. اختبارات الواجهة المطلوبة
==================================================

افتح وتحقق من الصفحات التالية:

1. الرئيسية /
يجب أن تعرض:
- بطاقة التاريخ.
- رسالة اليوم.
- مواقيت الصلاة.
- الصلاة القادمة.
- الوقت المتبقي.
- بطاقة "كم باقي على".
- العدادات المالية.
- المواعيد القادمة إن وجدت.

يجب ألا تظهر:
- لا توجد رسالة اليوم.
- لا توجد بيانات.
- لا توجد عدادات حالياً.

2. المال /finance
يجب أن يعرض:
- الرواتب.
- الدعم.
- الفواتير أو الأحداث المالية الأساسية.
- العدادات أو التواريخ القادمة.
- لا يكون فارغاً.

3. ستوري اليوم /story
يجب أن يعرض:
- التاريخ.
- رسالة اليوم.
- العدادات.
- محتوى قابل للنسخ أو الحفظ إن كان منفذاً.
- نفس مصدر البيانات المستخدم في الرئيسية.

4. الإشعارات /notifications إن كان المسار منفذاً
تحقق من:
- عدم وجود صفحة مكسورة.
- ظهور سجل أو حالة فارغة سليمة.
- عدم وجود خطأ API.

5. لوحة المالك /admin
تحقق من:
- الصفحة تفتح.
- ترى البيانات الأساسية أو أدوات إدارتها.
- لا يوجد خطأ 500.
- لا توجد شاشة بيضاء.
- إذا كان تعديل رسالة اليوم منفذاً، جرّب تعديلاً بسيطاً وتأكد أنه يظهر في الرئيسية وستوري اليوم.

6. الصفحات القانونية إن كانت موجودة:
- /privacy
- /terms
- /disclaimer
- /support

تحقق فقط أنها تفتح ولا تكسر التطبيق.

==================================================
7. اختبارات API المطلوبة
==================================================

افحص أن API لا يرجع أخطاء 500 للبيانات الأساسية.

تحقق من endpoints أو مسارات البيانات الخاصة بـ:
- daily_messages.
- prayer_times أو prayer_settings.
- financial_events.
- appointments أو public_events.
- story_templates.
- notifications إن وجدت.
- admin data إن وجدت.

المطلوب:
- لا توجد 500.
- لا توجد arrays فارغة للبيانات التي يجب أن تظهر.
- لا توجد أخطاء schema.
- لا توجد أخطاء missing table.
- لا توجد أخطاء missing column.

إذا ظهرت 500:
- أصلح السبب فقط.
- لا تغيّر التصميم.
- لا تضف ميزات.
- لا توسّع نطاق الإصلاح.

==================================================
8. اختبارات الاستمرارية بعد Reload
==================================================

نفّذ Reload للـ Preview ثم تحقق من:

- رسالة اليوم ما زالت ظاهرة.
- مواقيت الصلاة ما زالت ظاهرة.
- العدادات المالية ما زالت ظاهرة.
- مركز المال ما زال يعرض بيانات.
- ستوري اليوم ما زال يقرأ البيانات.
- /admin ما زال يعمل.
- لا تعود العبارات الفارغة بعد reload.

إذا اختفت البيانات بعد reload:
- المشكلة في persistence أو initialization.
- أصلحها بأضيق نطاق ممكن.
- لا تعتبر الحكم Publishable Preview.

==================================================
9. أوامر الفحص
==================================================

شغّل حسب المتاح في package.json فقط:

- pnpm install إذا احتاج.
- pnpm build أو npm run build حسب المشروع.
- npm run typecheck إن وجد.
- npm run lint إن وجد.
- npm run test إن وجد.

قواعد إلزامية:
- إذا كان الأمر غير موجود، لا تدّع أنك شغلته.
- إذا فشل أمر، أصلح سبب الفشل إن كان ضمن نطاق التحقق.
- إذا كان الفشل خارج النطاق، وثّقه واحكم Needs Fixes.
- لا تعتبر المهمة مكتملة بدون build ناجح.

==================================================
10. قاعدة الإصلاح المحدود
==================================================

إذا ظهر خطأ أثناء التحقق، أصلحه فقط إذا كان:

- خطأ build.
- خطأ TypeScript.
- خطأ API 500.
- صفحة مكسورة.
- بيانات أساسية لا تظهر.
- مشكلة reload تجعل البيانات تختفي.
- كسر في /admin يمنع التحقق.

ممنوع إصلاح أو تعديل:
- الشكل.
- الألوان.
- الثيم.
- مكونات جديدة.
- ميزات جديدة.
- مسارات جديدة غير مطلوبة.
- إعادة بناء Data Layer بالكامل.
- تغيير architecture بلا سبب.

أي إصلاح يجب أن يكون صغيراً، مباشراً، ومبرراً.

==================================================
12. تحديث التوثيق
==================================================

حدّث QA_REPORT.md أو التقرير المناسب بما يلي:

- أن المشروع تم نقله إلى حساب hrq.
- أن مشكلة الاعتماديات بعد ZIP تم إصلاحها.
- أن مشكلة البيانات الفارغة تم إصلاحها.
- ما البيانات التي ظهرت بعد الإصلاح.
- نتيجة build.
- نتيجة typecheck إن وجد.
- نتيجة Preview.
- هل توجد أخطاء API.
- هل توجد قيود متبقية.
- الحكم النهائي.

لا تكتب:
- Production Ready.
- Supabase متصل فعلياً إن لم يكن متصلاً.
- اختبارات لم تنفذها.
- نجاحات غير مؤكدة.
