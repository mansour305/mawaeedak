استمر من آخر Checkpoint. لا تعيد البناء من البداية.

المرحلة الحالية:
Phase 12I
Controlled Write Cutover Preparation + Safe Mutations Gate

الدور:
تصرف كفريق Data Layer Architect + Supabase Engineer + Frontend Reliability Engineer + QA Lead + Security Reviewer.
مهمتك الآن ليست تحويل كل الكتابة دفعة واحدة، وليست حذف API، وليست إعلان Production Ready.
مهمتك تجهيز وتنفيذ Write Cutover محدود وآمن جداً لعمليات كتابة مختارة فقط، مع الحفاظ على API كـ fallback وعدم كسر أي مرحلة سابقة.

==================================================
0. السياق الحالي
==================================================

تم اعتماد Phase 12H:
- Data Gateway موجود.
- Supabase Data Adapter موجود.
- Data Source Mode موجود.
- mode=api يعمل.
- mode=supabase_shadow يعمل.
- mode=supabase يعمل للقراءة.
- default = api.
- القراءة عبر Gateway مفعلة في:
  - CentersNewsPage عبر useGatewayNews
  - CentersJobsPage عبر useGatewayJobs
  - AccountPage Theme Picker عبر useGatewayThemes
  - StoryPage templates عبر useGatewayStoryTemplates

Supabase يحتوي 40 صفاً مطابقاً لـ API:
- daily_messages = 8
- story_templates = 2
- themes = 10
- news = 2
- jobs = 2
- appointments = 2
- financial_events = 8
- notifications = 3
- complaints = 3

القيد الحالي:
- الكتابة Mutations كلها ما زالت على API.
- CalendarPage/FinancePage/NotificationsPage/HomePage ما زالت على Orval.
- Admin pages ما زالت على Orval.
- mode=supabase يجب التأكد أنه يحترم env فعلاً قبل أي تحويل كتابة.

==================================================
1. الهدف التنفيذي
==================================================

نفّذ Write Cutover محدود وآمن جداً.

المطلوب:
1. تأكد أولاً أن dataSourceMode يحترم VITE_DATA_SOURCE_MODE فعلياً:
   - api
   - supabase_shadow
   - supabase
   ولا يرجع api دائماً إذا env=supabase.

2. لا تغيّر default. يجب أن يبقى default = api.

3. افحص جميع mutations الحالية وحددها في تقرير:
   - calendar appointments add/edit/delete
   - financial_events add/edit/delete
   - notifications send/mark-read/delete
   - daily_messages add/edit/delete/toggle
   - story_templates add/edit/delete/toggle
   - themes toggle/apply/admin update
   - news add/edit/delete
   - jobs add/edit/delete
   - complaints/contact forms
   - audit_logs

4. لا تحوّل كل mutations دفعة واحدة.

5. اختر أقل نطاق آمن لتحويل الكتابة أولاً، مع تبرير القرار.

6. النطاق المقترح الأول:
   - notifications mark-read فقط
   أو
   - story_templates admin CRUD إذا كان واضحاً
   أو
   - news/jobs admin CRUD إذا كان واضحاً

7. تجنب في هذه المرحلة:
   - appointments write
   - financial_events write
   لأنها مرتبطة بالرئيسية والتقويم والمال وستوري اليوم.

8. عند mode=api:
   - الكتابة تبقى API.

9. عند mode=supabase_shadow:
   - الكتابة تبقى API.
   - لا تكتب في Supabase.

10. عند mode=supabase:
   - فقط النطاق المختار يكتب إلى Supabase.
   - بقية mutations تبقى API.
   - وثّق ذلك بوضوح.

11. لا تحذف API.
12. لا تحذف PostgreSQL.
13. لا تضغط Publish.
14. حدّث QA_REPORT والوثائق.

3. ممنوعات صارمة
==================================================

ممنوع:
- لا تحول كل mutations مرة واحدة.
- لا تجعل Supabase default.
- لا تحذف API endpoints.
- لا تستخدم service_role في الواجهة.
- لا تضع أسرار في الكود.
- لا تستخدم DROP/TRUNCATE/DELETE.
- لا تنقل بيانات جديدة خارج mutation الاختبارية المختارة.
- لا تضغط Publish.
- لا تدّع Production Ready.
- لا تكسر أي صفحة.
- لا تكسر RLS.
- لا تكتب إلى API وSupabase معاً لنفس العملية إلا إذا كان dual-write موثقاً ومقصوداً ومؤقتاً، والأفضل تجنبه الآن.

==================================================
4. بروتوكول التشخيص قبل التنفيذ
==================================================

قبل أي تعديل:

1. افحص dataSourceMode.ts وتأكد عملياً أنه يقرأ env عند ضبط VITE_DATA_SOURCE_MODE.
2. افحص dataGateway.ts.
3. افحص supabaseData.ts.
4. افحص كل mutations في المشروع.
5. صنّف mutations حسب الخطورة:
   - Low risk
   - Medium risk
   - High risk

6. حدّد النطاق الأول للتحويل.
7. وثّق سبب اختياره.
8. افحص RLS لذلك الجدول.
9. افحص هل المستخدم current session يملك صلاحية الكتابة.
10. افحص هل shape البيانات مناسب.
11. افحص fallback behavior.
12. لا تبدأ التنفيذ قبل اكتمال التشخيص.

==================================================
5. سياسة التنفيذ
==================================================

نفذ بأقل تغيير آمن:

1. أضف write functions في Supabase adapter فقط للنطاق المختار.
2. أضف gateway mutation function فقط للنطاق المختار.
3. اربط صفحة واحدة أو flow واحد فقط إذا كان آمناً.
4. mode=api:
   يستخدم API.
5. mode=supabase_shadow:
   يستخدم API.
6. mode=supabase:
   يستخدم Supabase للنطاق المختار فقط.
7. عند فشل Supabase write:
   - لا تنفذ fallback صامت إلى API إذا كان ذلك يسبب divergence.
   - اعرض خطأ واضح أو وثق القرار.
8. لا تغيّر بقية mutations.

7. الاختبارات الإلزامية
==================================================

1. mode=api:
