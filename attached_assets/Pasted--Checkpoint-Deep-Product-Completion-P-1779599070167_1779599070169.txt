==================================================
1. الهدف التنفيذي
==================================================

حوّل ستوري اليوم من عرض بيانات إلى أداة عملية قابلة للاستخدام.

المطلوب تنفيذ أو التحقق من التالي:

1. ستوري اليوم يقرأ البيانات من Data Layer وليس نصوصاً ثابتة داخل component.
2. يعرض:
   - التاريخ الهجري.
   - التاريخ الميلادي.
   - اليوم.
   - رسالة اليوم.
   - عدادات الرواتب والدعم والفواتير أو العناصر المالية المتاحة.
   - مواقيت الصلاة إن كانت ضمن تصميم الستوري الحالي.
3. وجود معاينة واضحة للستوري.
4. إمكانية تعديل نص رسالة الستوري محلياً قبل النسخ/الحفظ.
5. إمكانية اختيار العناصر الظاهرة في الستوري:
   - التاريخ.
   - رسالة اليوم.
   - الصلاة.
   - الرواتب.
   - الدعم.
   - الفواتير.
6. إمكانية اختيار قالب أو نمط عرض إن كانت البنية الحالية تسمح.
7. زر نسخ نص الستوري.
8. زر حفظ داخلي للستوري.
9. زر مشاركة إن كانت البيئة تدعم Web Share API، أو fallback واضح للنسخ إذا لم تدعم.
10. بقاء إعدادات الستوري أو آخر نسخة محفوظة بعد reload إذا كان Data Layer/fallback الحالي يدعم ذلك.
11. عدم كسر بيانات الرئيسية أو المال أو التقويم.
12. تحديث QA_REPORT بنتيجة المرحلة.

2. خارج النطاق
==================================================

هذه المرحلة ليست:
- توليد صورة حقيقية للستوري إذا كانت البيئة لا تدعم ذلك.
- رفع صور خارجية.
- نظام نشر اجتماعي حقيقي.
- جدولة خارجية للنشر.

- Auth/RLS.
- Firebase.

- Publish.

إذا تعذر توليد صورة:
وفر معاينة تصميمية + نسخ نص + حفظ داخلي.
لا تترك الميزة فارغة.

==================================================
3. ممنوعات صارمة
==================================================

ممنوع:
- لا تعيد بناء المشروع من البداية.
- لا تغيّر Routing.
- لا تغيّر API إلا إذا كان endpoint خاص بالستوري مكسوراً ويحتاج إصلاحاً محدوداً.
- لا تغيّر Data Layer جذرياً.
- لا تغيّر Theme العام إلا لإصلاح كسر بسيط داخل صفحة الستوري.
- لا تضف Supabase الآن.
- لا تضف Firebase الآن.
- لا تضغط Publish.
- لا تجعل الستوري نصاً hardcoded داخل component.
- لا تجعل زر نسخ/حفظ/مشاركة بلا وظيفة.
- لا تدّع مشاركة خارجية إذا Web Share API غير متاح.
- لا تدّع توليد صورة إذا غير منفذ.
- لا تدّع نجاح build إن لم يتم تشغيله.
- لا تغيّر الحكم إلى Production Ready.

==================================================
4. بروتوكول التشخيص قبل التنفيذ
==================================================

قبل التعديل، افحص:

1. أين صفحة ستوري اليوم؟
2. هل توجد story service أو Data Layer خاص بالستوري؟
3. من أين تأتي رسالة اليوم؟
4. من أين تأتي العدادات المالية؟
5. من أين يأتي التاريخ؟
6. هل يوجد قالب ستوري؟
7. هل توجد أزرار نسخ/حفظ/مشاركة حالياً؟
8. هل الأزرار تعمل فعلاً؟
9. هل النص الحالي hardcoded داخل component؟
10. هل ستوري اليوم مرتبط بالرئيسية والمال من نفس Data Layer؟
11. هل يوجد حفظ داخلي لآخر ستوري أو إعداداته؟
12. هل توجد أخطاء API أو console حالية؟

بعد التشخيص:
- إذا كانت الوظيفة موجودة جزئياً، أكمل الناقص فقط.
- إذا كانت موجودة لكنها تعتمد على نص ثابت، اربطها بالـ Data Layer.
- إذا كانت أزرار النسخ/الحفظ/المشاركة شكلية، اجعل لها أثر واضح.
- لا تلمس أجزاء خارج النطاق.

==================================================
5. قاعدة الإصلاح الآمن
==================================================

نفذ بأقل تغيير آمن.

القواعد:
- ستوري اليوم يقرأ من daily message service.
- ستوري اليوم يقرأ العدادات من financial event/countdown service.
- التاريخ من date utils أو المصدر الحالي المعتمد.
- لا تكرر بيانات المال داخل component.
- لا تكرر رسالة اليوم داخل component.
- أي تعديل نصي مؤقت يجب أن يكون واضحاً كمحتوى الستوري الحالي.
- أي حفظ داخلي يجب أن يمر عبر service/fallback الحالي قدر الإمكان.
- أي عملية نجاح أو فشل تعرض feedback واضحاً.
- لا تكسر RTL.
- لا تكسر Mobile layout.
- لا تكسر Visual Polish المعتمد.

6. مصدر الحقيقة
==================================================

مصدر الحقيقة لهذه المرحلة:

- daily_messages أو ما يعادلها في Data Layer.
- financial_events / countdown أو ما يعادلها في Data Layer.
- date utils للتاريخ الهجري والميلادي.
- story_templates أو story service إن كانت موجودة.

ممنوع:
- تكرار رسالة اليوم داخل component.
- تكرار العدادات المالية داخل component.
- قراءة ستوري اليوم من مصدر مختلف عن الرئيسية والمال.
- كتابة أرقام العدادات يدوياً.
- جعل النص النهائي ثابتاً غير مرتبط بالبيانات.

المطلوب:
- إذا تغيّرت رسالة اليوم من Data Layer، يظهر أثرها في ستوري اليوم.
- إذا تغيّرت العدادات المالية، يظهر أثرها في ستوري اليوم.
- إذا عدّل المستخدم نص الستوري في الواجهة، تظهر المعاينة فوراً.
- النسخ ينسخ النص الظاهر فعلياً.
- الحفظ يحفظ النص/الإعدادات فعلياً قدر الإمكان.

8. المتطلبات التفصيلية
==================================================

أ. صفحة ستوري اليوم:
- معاينة تصميمية واضحة.
- نص قابل للتعديل.
- اختيار العناصر الظاهرة.
- زر نسخ.
- زر حفظ.
- زر مشاركة أو fallback للنسخ.
- حالة فارغة عند عدم توفر بيانات.
- حالة خطأ واضحة عند فشل التحميل.

ب. النسخ:
- ينسخ نص الستوري الحالي كما يظهر بعد التعديل.
- يظهر feedback بعد النسخ.

ج. الحفظ:
- يحفظ آخر نسخة أو إعدادات الستوري حسب البنية الحالية.
- يظهر feedback بعد الحفظ.
- يبقى بعد reload إن كان المصدر الحالي يدعم ذلك.

د. المشاركة:
- إذا Web Share API متاح: استخدمه.
- إذا غير متاح: استخدم fallback copy.
- لا تدّع مشاركة خارجية إذا لم تعمل.

هـ. القوالب:
- إذا توجد story_templates، فعّل اختيار قالب بسيط.
- إذا لا توجد، لا تبني نظام قوالب ضخم الآن.
- وفر قالباً واحداً عملياً على الأقل مع إمكانية التعديل.
