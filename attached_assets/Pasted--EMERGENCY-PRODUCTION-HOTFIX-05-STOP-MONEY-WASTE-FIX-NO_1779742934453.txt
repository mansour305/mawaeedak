==================================================
1. إصلاح مشكلة الإشعارات: العداد 2 والصفحة فارغة
==================================================

المشكلة:
TopBar/Badge يعرض أن هناك إشعارين، لكن عند فتح NotificationsPage لا توجد إشعارات.

هذا يعني أحد هذه الأخطاء:
- unread-count من مصدر مختلف عن قائمة الإشعارات.
- count يقرأ من API والقائمة من Supabase أو العكس.
- fallback يخلط بين API/Supabase.
- user_id mismatch.
- cache stale.
- notifications list filtered بطريقة مختلفة عن count.
- count لا يُصفّر عند عدم وجود نتائج.
- localStorage أو demo data ما زال يؤثر.
- query invalidation ناقص.

المطلوب:
1. تتبع مصدر العداد:
   - TopBar
   - useGatewayUnreadCount
   - gwGetUnreadNotificationsCount
   - API /api/notifications/unread-count
   - Supabase count
2. تتبع مصدر القائمة:
   - NotificationsPage
   - useGatewayNotifications
   - gwGetNotifications
   - API /api/notifications
   - Supabase select
3. اجعل العداد والقائمة من نفس مصدر الحقيقة.
4. القاعدة النهائية:
   - إذا القائمة للمستخدم الحالي فارغة، العداد يجب أن يكون 0.
   - إذا العداد 2، يجب أن تظهر إشعاران فعلياً في الصفحة.
5. في mode=supabase:
   - count يستخدم نفس فلتر قائمة notifications:
     user_id = current user
     is_read = false
   - لا fallback صامت يعطي count من API بينما list من Supabase.
6. في mode=api:
   - count والقائمة من API مع نفس المستخدم/السياق.
7. عند delete/mark-read/mark-all:
   - invalidate notifications
   - invalidate unread-count
   - لا يبقى badge قديم.
8. إذا لا توجد session:
   - لا تعرض badge وهمي.
   - count = 0 أو guest-safe.
9. احذف أي demo notification count أو hardcoded badge.

اختبارات إلزامية:
- زائر بدون session: badge لا يظهر أو = 0.
- user له 0 إشعارات: badge = 0 والصفحة empty.
- user له 2 unread: badge = 2 والصفحة تعرض 2.
- mark one read: badge ينقص.
- mark all read: badge = 0.
- delete notification: badge والقائمة يتحدثان.
- refresh بعد كل حالة لا يرجع count قديم.
- لا يوجد اختلاف API/Supabase.

2. إصلاح مواقيت الصلاة حسب الموقع الحقيقي
==================================================

المشكلة:
مواقيت الصلاة للمناطق خاطئة.

المطلوب:
مواقيت الصلاة يجب أن تعتمد على:
- GPS lat/lng إذا سمح المستخدم.
- المدينة اليدوية إذا رفض أو لم يسمح.
- fallback واضح فقط عند عدم وجود بيانات.

افحص:
- useLocationPrefs
- AccountPage location section
- HomePage prayer strip
- prayer-times API
- أي city mapping / Haversine / timezone
- 12h/24h formatting

القواعد:
1. عند السماح بالموقع:
   - استخدم navigator.geolocation.getCurrentPosition.
   - احفظ latitude/longitude بدقة.
   - احفظ timezone من Intl.DateTimeFormat().resolvedOptions().timeZone.
   - source = gps.
2. API /api/prayer-times يجب أن يقبل:
   - lat
   - lng
   - timezone
   - city fallback
3. لا تعتمد على أقرب مدينة فقط إذا lat/lng موجود.
4. عند رفض الموقع:
   - اعرض اختيار المدينة يدوياً.
   - استخدم إحداثيات المدينة المختارة من قائمة دقيقة.
5. لا تكون الرياض default إلا إذا:
   - لا GPS
   - لا اختيار يدوي
   - لا بيانات محفوظة
6. HomePage يجب أن يوضح:
   - "حسب موقعك"
   أو
   - اسم المدينة المختارة
7. زر "تحديث موقعي" يجب أن يجدد lat/lng ويعيد جلب الصلاة.
8. تغيير المدينة يغير مواقيت الصلاة فوراً.
9. صيغة 12/24 لا تنكسر.
10. StoryPage إن عرضت الصلاة يجب أن تستخدم نفس المصدر.

قائمة المدن السعودية يجب مراجعتها:
- الرياض
- مكة
- المدينة
- جدة
- الدمام
- الخبر
- الظهران
- الطائف
- أبها
- خميس مشيط
- بريدة
- حائل
- تبوك
- جازان
- نجران
- الباحة
- سكاكا
- عرعر
- ينبع
- الجبيل
- الأحساء

اختبارات إلزامية:
- mock GPS الرياض → مواقيت الرياض.
- mock GPS جدة → مواقيت جدة.
- mock GPS مكة → مواقيت مكة.
- اختيار يدوي جدة → تتغير عن الرياض.
- اختيار يدوي الدمام → تتغير عن جدة.
- رفض الموقع → تظهر المدينة اليدوية.
- refresh يحافظ على الاختيار.
- لا تظهر مواقيت الرياض لكل المدن.
